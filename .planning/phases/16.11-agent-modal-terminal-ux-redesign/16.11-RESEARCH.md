## Summary

Phase 16.11 has four distinct sub-problems — contrast/legibility, timeline sort order, event card rendering, and WebSocket terminal input — each with a clear implementation path. Three of the four are pure UI changes with no daemon work; the fourth (terminal input) requires a new WS message type and a daemon handler.

**Root cause of black text on dark background:** `bg-background` in Tailwind v4 maps to `--color-background: oklch(0.2 0.02 255)` — a very dark colour. The `text-foreground` class maps to `--color-foreground: oklch(0.96 0.01 255)` — near-white. So heading/title text is actually fine. The visual bug is almost certainly inside the tab content panels rendered inside the modal. `TimelinePanel`'s `InlineDetail` component uses `text-muted-foreground` for JSON `<pre>` blocks; in some browser rendering contexts the `<pre>` may fall back to the browser's default text colour (black) if Tailwind's CSS variable isn't resolved inside jsdom or the live app. The `InlineDetail` fallback branch (`JSON.stringify(event, null, 2)`) uses no explicit text class, inheriting from the `base` class which has no foreground set — any parent that resets to dark text would cause black-on-dark.

**Timeline sort:** `filteredEvents` is derived directly from `events[sessionId]` which is stored in insertion/sequence order (oldest first). The fix is a single `[...filteredEvents].reverse()` before the render map, preserving the existing `jumpTargets` logic which must continue to work on the unfiltered, unmodified `events` array.

**Terminal input:** `sendWsMessage(msg: object)` already exists in `useSessionEvents.ts` (line 110-114). The daemon WS `handlers.ts` currently only handles `type === 'approval_decision'`. A new message type (e.g. `{ type: 'terminal_input', sessionId, text }`) requires a handler in `handlers.ts`. The daemon does not currently have a pipe/stdin channel to running Claude sessions (Claude hooks are HTTP-only), so the terminal input for Claude sessions will need to be stored/surfaced but cannot be forwarded to the agent in Phase 16.11 — the planner should scope this to UI + daemon WS echo only (daemon logs the command, no actual agent forwarding).

**Primary recommendation:** Fix contrast by auditing all color classes in TimelinePanel/InlineDetail fallback paths and adding explicit `text-[var(--color-muted-foreground)]` or `text-foreground` to every text element; reverse `filteredEvents` before render; build per-type EventCard components using existing CSS tokens; wire terminal input bar through `sendWsMessage` to a new `terminal_input` daemon handler.

### Pattern 1: Contrast Fix — Explicit Foreground on Every Text Node

**What:** Tailwind v4 does not inject a global `color` reset into arbitrary components. `<pre>` and plain `<div>` inside Radix Dialog inherit from Radix's own Portal/Content overlay which may reset colour. Every leaf text element must carry an explicit text colour utility.

**When to use:** Any element inside `InlineDetail` or the fallback branch of the event renderer.

**Token map for the terminal theme:**
```
Primary text (legible body):   text-[var(--color-foreground)]        oklch(0.96 0.01 255) — near-white
Secondary/dim text:            text-[var(--color-muted-foreground)]   oklch(0.80 0.02 250) — 80% lightness
Accent/highlight text:         text-[var(--color-cockpit-cyan)]       oklch(0.75 0.18 195)
Warning text:                  text-[var(--color-cockpit-amber)]      oklch(0.75 0.16  75)
Danger text:                   text-[var(--color-cockpit-red)]        oklch(0.60 0.22  25)
Dim/inactive:                  text-[var(--color-cockpit-dim)]        oklch(0.35 0.04 220) — DO NOT use for body text
Panel surface bg:              bg-[var(--color-panel-surface)]        oklch(0.185 0.025 250)
```

WCAG AA requires contrast ratio >= 4.5:1 for normal text.
- `--color-foreground` (L=0.96) on `--color-panel-surface` (L=0.185): HIGH contrast — PASS
- `--color-muted-foreground` (L=0.80) on `--color-panel-surface` (L=0.185): PASS
- `--color-cockpit-dim` (L=0.35) on `--color-panel-surface` (L=0.185): MARGINAL — use only for intentional de-emphasis, never for primary content

**The fallback InlineDetail branch** (`JSON.stringify(event, null, 2)`) currently has no text colour class — this is the black-text bug. Fix: add `text-[var(--color-muted-foreground)]` to the `<pre>`.

### Pattern 2: Timeline Newest-First Sort

**What:** Reverse `filteredEvents` array before the render loop. The `jumpTargets` logic must NOT be reversed because it uses original `events` indices for `scrollIntoView` via `rowRefs`.

**When to use:** The `filteredEvents` render map only.

**Key detail:** `rowRefs` keys on `idx` from the original `filteredEvents.map((event, idx) => ...)`. After reversing, the visual order changes but `idx` still runs 0..N-1. The `jumpTargets` stores original indices from `events.map((e, i) => ...)` (unfiltered, unreversed). Jump scrolling calls `rowRefs.current.get(next.i)` — this will break if `idx` in the render map no longer corresponds to position `i` in the unfiltered array. **Fix:** Continue using `idx` as the rowRef key for the reversed display order, but the jump `scrollIntoView` should be re-assessed — the jump system works on sequence numbers, not array positions. Consider keying rowRefs by `sequenceNumber` instead of `idx` for correctness.

```typescript
// In TimelinePanel.tsx — replace the filteredEvents.map call:
const displayEvents = [...filteredEvents].reverse()

// ...then map over displayEvents instead of filteredEvents
{displayEvents.map((event, idx) => { ... })}
```

### Pattern 4: Terminal Input Bar

**What:** A fixed-to-bottom input bar inside InstancePopupHub (not inside TimelinePanel) that sends a `terminal_input` WS message to the daemon.

**Architecture:**
- Input bar lives in `InstancePopupHub.tsx` below the `<Tabs.Root>` — always visible regardless of which tab is active
- Uses `sendWsMessage` from `useSessionEvents.ts` — already exported
- Disabled when `wsStatus !== 'connected'` (read from `useStore`)
- Message format: `{ type: 'terminal_input', sessionId: string, text: string }`
- Daemon handler in `handlers.ts`: receives `terminal_input` message, logs it, emits acknowledgement (no actual agent forwarding in Phase 16.11 — Claude sessions have no stdin pipe)

**Input bar chrome (terminal aesthetic):**
```tsx
<div className="shrink-0 border-t border-[var(--color-cockpit-dim)]/40 bg-[var(--color-panel-surface)] flex items-center gap-2 px-3 py-2">
  <span className="text-[var(--color-cockpit-cyan)] text-[10px] font-[var(--font-mono-data)] select-none">
    &gt;_
  </span>
  <input
    type="text"
    value={inputText}
    onChange={(e) => setInputText(e.target.value)}
    onKeyDown={(e) => { if (e.key === 'Enter') handleSend() }}
    disabled={wsStatus !== 'connected'}
    placeholder="enter command..."
    className="flex-1 bg-transparent text-[var(--color-foreground)] text-[11px] font-[var(--font-mono-data)]
               placeholder:text-[var(--color-cockpit-dim)] outline-none disabled:opacity-40"
  />
  <button onClick={handleSend} disabled={!inputText.trim() || wsStatus !== 'connected'}
    className="cockpit-btn text-[9px] py-0.5 disabled:opacity-30">
    SEND
  </button>
</div>
```

### Anti-Patterns to Avoid

- **Using `text-muted-foreground` for the `--color-cockpit-dim` intent:** `data-readout-dim` uses dim (L=0.35) which fails WCAG AA — use `muted-foreground` (L=0.80) instead for any readable text
- **Mutating `events` array for display order:** Never sort/reverse in place — always `[...filteredEvents].reverse()` to preserve store integrity
- **Keying rowRefs by array index after reversal:** After reversing displayEvents, the render index `idx` no longer matches jump positions — key rowRefs by `sequenceNumber` instead
- **Adding terminal input inside TimelinePanel:** The terminal bar should live in `InstancePopupHub` so it persists across tab changes and appears as global modal chrome
- **Forwarding terminal input to Claude agent in Phase 16.11:** Claude sessions communicate via HTTP hooks only — there is no stdin pipe. Scope terminal input to WS message logging only for this phase.

## Common Pitfalls

### Pitfall 1: Radix Dialog Content inherits browser default text colour

**What goes wrong:** Radix Dialog.Content renders in a Portal. The portal element is appended to `<body>`. In some environments, `<body>` has `color: black` (browser default). If a descendant element doesn't specify a text colour, it inherits black-on-dark. The `bg-background` in InstancePopupHub sets the background to dark, but any child element without explicit text colour can inherit black from `<body>`.

**Why it happens:** Tailwind v4 uses `@theme` to define CSS variables, but does not necessarily inject a global `color` reset on `*` or `body`. If the `:root` or `body` doesn't have `color: var(--color-foreground)`, children inherit browser default.

**How to avoid:** Check `packages/ui/src/index.css` or the root app component — confirm `body { color: var(--color-foreground); }` exists, OR ensure every leaf text element carries an explicit text colour class. The safer fix is both: add a `body { color: var(--color-foreground); background: var(--color-background); }` rule to index.css AND audit all text elements.

**Warning signs:** `<pre>` blocks, `<span>` inside `InlineDetail` fallback, any element without explicit `text-*` class.

### Pitfall 2: Jump-to-next breaks after timeline reversal

**What goes wrong:** `jumpNext()` calls `rowRefs.current.get(next.i)` where `next.i` is the index into the unfiltered `events` array. After reversing `filteredEvents`, the DOM rows are in reverse order but `rowRefs` is still keyed by the old array index. A `scrollIntoView` call on index 3 will scroll to the 4th-from-newest event instead of the 4th event in original order.

**Why it happens:** `rowRefs` is keyed by `idx` from `filteredEvents.map((event, idx) => ...)`. The reversal makes `idx=0` correspond to the newest event, while the jump system thinks `idx=0` is the oldest.

**How to avoid:** Re-key `rowRefs` by `sequenceNumber` rather than by `idx`. Update jump target storage and lookup to also use `sequenceNumber`.

**Warning signs:** Jump buttons work but scroll to wrong event after timeline reversal.

### Pitfall 3: `sendWsMessage` silently no-ops when disconnected

**What goes wrong:** `sendWsMessage` checks `ws?.readyState === WebSocket.OPEN` and silently does nothing if disconnected. User types a command and presses Enter — nothing happens, no error.

**Why it happens:** Designed for fire-and-forget; silence is intentional to avoid crashes.

**How to avoid:** Read `wsStatus` from Zustand store in the terminal input component. Disable the input when `wsStatus !== 'connected'`. Show a small status indicator near the input (connected = cyan dot, disconnected = dim/red).

**Warning signs:** User reports command appeared to be sent but nothing happened.

### Pitfall 4: Daemon has no `terminal_input` handler

**What goes wrong:** UI sends `{ type: 'terminal_input', sessionId, text }` over WS. `handlers.ts` only handles `approval_decision` — the `terminal_input` message falls through the switch silently.

**Why it happens:** The daemon WS handler has a single `if` check, no default catch-all logging.

**How to avoid:** Add a `terminal_input` case to `handlers.ts`'s `ws.on('message')` handler. For Phase 16.11, log the command and emit a `terminal_ack` event back to the client — no actual agent forwarding.

**Warning signs:** WS message sent, no response, no log output from daemon.

### Pitfall 5: `[...filteredEvents].reverse()` in render causes test failures

**What goes wrong:** Existing tests in `TimelinePanel.test.tsx` assert that events render in original order (oldest first). After reversing, tests that check "first visible event" will fail.

**Why it happens:** 8+ existing tests assert rendering order implicitly.

**How to avoid:** Update the test suite alongside the implementation. Tests for "newest first" need to check that the LAST event in the array renders at the TOP of the list.

## Open Questions

1. **Should the terminal input bar forward commands to Claude agents?**
   - What we know: Claude communicates via HTTP hooks (no stdin). Codex is a spawned child process with stdin pipe — CodexAdapter.proc exists but has no `sendInput` method.
   - What's unclear: Whether Phase 16.11 should add stdin forwarding for Codex, or defer to a future phase.
   - Recommendation: Scope to WS logging only in Phase 16.11. Add a `// TODO Phase X: route to CodexAdapter.sendInput()` comment. Avoid scope creep.

2. **Should `jumpTargets` indices be updated to work with newest-first display?**
   - What we know: Current jump logic uses array indices into the unfiltered events array.
   - What's unclear: Whether the user expects "Next Approval" to scroll to the chronologically-next or the display-next approval.
   - Recommendation: Semantically, "Next Approval" should mean chronologically-next (older in the reversed view). Re-key rowRefs by sequenceNumber. Update jumpNext/jumpPrev to work with sequenceNumber-keyed refs.

3. **Is `body { color: var(--color-foreground) }` already defined somewhere?**
   - What we know: `index.css` does not have a `body` rule. The `@theme` block only defines custom properties.
   - What's unclear: Whether the Vite/Tailwind base styles inject a body reset.
   - Recommendation: Add an explicit `body` rule in Wave 0 as insurance. Audit in browser devtools during QA checkpoint.
