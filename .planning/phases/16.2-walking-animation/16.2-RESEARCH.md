# Phase 16.2: Walking Animation & Sprite Quality — Research

**Researched:** 2026-04-12
**Domain:** Canvas sprite animation, game loop frame stepping, sprite sheet layout
**Confidence:** HIGH

---

## Summary

Phase 16.2 adds walk-cycle animation to the player character and state-cycling animation to NPC agents. The sprite sheets are already fully generated (Phase 13) with multi-frame walk and state rows — the frames simply have never been displayed because both player and NPC rendering hardcode `col = 0`. The full fix is surgical: add `animTime: number` to `GameState.player`, advance it inside `movePlayer()`, and compute `col` from it in the two render sites (`OfficePage.tsx` for player, `AgentSprite.tsx` for NPCs).

No new libraries are needed. The entire phase is pure TypeScript / Canvas 2D API work on files that already exist. PixelLab sprite improvement is a conditional plan gated on visual QA after animation is wired — the manifests confirm walk frames already exist, so regeneration is unlikely to be needed.

**Primary recommendation:** Implement time-based `animTime` accumulation in `movePlayer()`, pass `tick` into `drawAgentSprite()`, and wire column selection at both render sites. Do not regenerate sprites unless visual QA reveals a real quality problem.

## Architecture Patterns

### Pattern 1: Time-based frame stepping (player)

Use accumulated `animTime` (milliseconds) modulo total cycle duration.

**Why time-based, not tick-based for player:** At 60fps with 4 frames, tick-based at rate 1 tick/frame = 15ms per frame (66fps for the walk anim). Too fast. Rate 8 ticks/frame = 133ms per frame ≈ 7.5fps — reasonable but inexact. Time-based at 150ms/frame is frame-rate independent and precise.

```ts
// Source: established top-down RPG pattern, verified against frame counts
export const WALK_FRAME_DURATION_MS = 150  // ~6.6 fps walk cycle
export const WALK_FRAME_COUNT = 4          // idle row has 4 frames

// Inside movePlayer():
const isMoving = dx !== 0 || dy !== 0
if (isMoving) {
  player.animTime += deltaMs
} else {
  player.animTime = 0  // snap to rest frame on key release
}

// At render site:
const col = Math.floor(player.animTime / WALK_FRAME_DURATION_MS) % WALK_FRAME_COUNT
```

**Reset on stop is critical:** if animTime keeps running, the character freezes on a random mid-stride frame when the player releases keys. Resetting to 0 always shows frame 0 (standing pose) when idle.

### Pattern 2: Tick-based frame stepping (NPCs)

NPCs have no movement (Phase 17) but must cycle their state animations (blocked, completed, failed).

```ts
const NPC_TICKS_PER_FRAME = 8  // advance frame every 8 game ticks ≈ 133ms @ 60fps
const frameCount = NPC_FRAME_COUNTS[animState]
const col = Math.floor(tick / NPC_TICKS_PER_FRAME) % frameCount
```

Tick-based is fine for NPCs because:
1. They are always animating (no start/stop to manage).
2. Using the shared `gameState.tick` avoids per-NPC state.
3. Slight inaccuracy at non-60fps is acceptable for background characters.

### NPC frame count table

```ts
const NPC_FRAME_COUNTS: Record<AnimationState, number> = {
  idle: 4,
  blocked: 8,
  completed: 9,
  failed: 7,
}
```

### Pattern 3: Extending DrawAgentSpriteOptions

`drawAgentSprite()` needs `tick` added to its options interface:

```ts
export interface DrawAgentSpriteOptions {
  ctx: CanvasRenderingContext2D
  session: SessionRecord
  lastEvent: NormalizedEvent | undefined
  position: { x: number; y: number }
  direction?: Direction
  imageCache: Map<string, HTMLImageElement>
  tick: number   // NEW — pass gameState.tick from render loop
}
```

The call site in `OfficePage.tsx` render() already has `gameState.tick` available.

### Anti-Patterns to Avoid

- **Tick-based for player:** Results in frame-rate-dependent animation speed. Use time-based.
- **animTime never resets:** Player snaps to mid-stride frame at idle. Always reset to 0 when not moving.
- **Per-NPC animTime state:** Adds unnecessary state to gameState.npcs. Use the shared tick counter.
- **Reading manifests at runtime for frame counts:** Adds async complexity. Hardcode from the locked Phase 13 values — they are guaranteed identical across all 10 character types.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sprite frame stepping | Custom animation scheduler | `Math.floor(animTime / duration) % count` | One-liner; no scheduler needed at this frame rate |
| NPC independent timers | Per-NPC timestamp tracking | Shared `gameState.tick` | Tick already increments every rAF; no extra state |
| Frame easing | Interpolation between frames | Direct frame snap | Pixel art looks better with crisp frame snaps, not blends |

## Common Pitfalls

### Pitfall 1: makePlayer() helper in tests not updated

**What goes wrong:** PlayerInput.test.ts has `function makePlayer(x = 0, y = 0, direction = 'south')` returning `{ x, y, direction }`. After adding `animTime` to the player type, TypeScript will reject test calls that pass the old object to `movePlayer()`.

**How to avoid:** Update `makePlayer()` to include `animTime = 0` as a fourth optional parameter in the same commit that adds `animTime` to the type. Failing to do so causes a cascade of test compilation errors.

### Pitfall 2: GameState singleton not initialized with animTime

**What goes wrong:** The exported `gameState` singleton is declared without `animTime`, so the initial value omits the field. TypeScript type error, and also runtime `undefined` arithmetic (`undefined + 150 = NaN`).

**How to avoid:** Update both the interface and the singleton initializer in `GameState.ts` simultaneously:
```ts
player: { x: 1472, y: 1376, direction: 'south', animTime: 0 }
```

### Pitfall 3: animTime overflows (no upper bound)

**What goes wrong:** Over a long session, `animTime` grows to millions. `Math.floor(animTime / 150) % 4` still works correctly due to modulo, but the number precision degrades after ~9 quadrillion ms (not a practical risk). More relevantly, `animTime` should never go negative.

**How to avoid:** No action needed for overflow — modulo handles it correctly. Just ensure the reset-to-0 path is always hit when not moving.

### Pitfall 4: drawAgentSprite callers break when tick is added

**What goes wrong:** `tick` added as required field to `DrawAgentSpriteOptions` but `OfficePage.tsx` call site is not updated. TypeScript compilation error.

**How to avoid:** Make `tick` required (not optional) in the interface — forces all callers to update. There is only one caller (`OfficePage.tsx` render loop), so this is a two-line fix.

// Source: verified pattern from 16.2-PLAN.md, consistent with spriteStates.ts frame counts
export const WALK_FRAME_DURATION_MS = 150
export const WALK_FRAME_COUNT = 4

export function movePlayer(
  player: { x: number; y: number; direction: string; animTime: number },
  keys: ReadonlySet<string>,
  deltaMs: number,
): void {
  // ... existing dx/dy/bounds logic unchanged ...

  const isMoving = dx !== 0 || dy !== 0
  if (isMoving) {
    player.animTime += deltaMs
  } else {
    player.animTime = 0
  }
}

## Open Questions

1. **PixelLab sprite quality**
   - What we know: Frames were generated in Phase 13 with `animate_character`; manifests confirm frame counts > 1; sprites have never been animated in the UI.
   - What's unclear: Whether the generated walk frames form a believable gait cycle (can't verify without running the animation).
   - Recommendation: Implement 16.2-01 through 16.2-03 first, then do a manual QA pass. Only execute 16.2-04 if walk frames are visually broken. Budget: ~1438 unspent images; regenerating all 10 characters costs ~320 calls.

2. **NPC animation speed coordination**
   - What we know: `NPC_TICKS_PER_FRAME = 8` gives ~133ms/frame at 60fps.
   - What's unclear: Whether all NPCs should animate at the same speed or state-dependent speeds (e.g., `failed` slower/faster than `blocked`).
   - Recommendation: Use uniform 8 ticks/frame for all states in this phase; tweak if visual QA reveals a problem.
