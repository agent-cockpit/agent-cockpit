**Researched:** 2026-04-12
**Domain:** 2D pixel art sprite animation, walk cycle timing, direction-aware row selection, animation state machine
**Confidence:** HIGH

## Summary

Phase 16.2 implemented the mechanical wiring for player walking animation (animTime accumulation, col formula, row switching). Phase 16.4 generated proper walk sprite frames and added them at row group offset 32. What Phase 16.5 must fix is the behavioral feel: the current code has several issues that make walking feel wrong even after those structural fixes land.

**The core problems identified from reading the actual code (HIGH confidence):**

1. **Wrong row for walking.** `OfficePage.tsx` line 115 still uses `STATE_ROW_OFFSET.blocked` (= 8) for walking. Phase 16.4 plans to fix this to `STATE_ROW_OFFSET.walk` (= 32), but Phase 16.5 must ensure it is done and verify it visually.

2. **No direction persistence on idle.** When the player stops, `direction` is correct (last direction used), but `animTime` resets to 0 immediately. Frame 0 of the idle row-group is shown. If idle frame 0 is a south-facing pose but the player last moved north, the wrong idle frame shows. The idle row must use the player's last direction, not a hardcoded row.

3. **`isMoving` check is pre-normalization.** `PlayerInput.ts` line 98: `const isMoving = up || down || left || right`. This is correct — but `animTime` advances even on diagonal moves where actual displacement may be near-zero due to opposite keys cancelling. If the user holds W+S simultaneously, `up=true, down=true`, `dy=0`, but `animTime` still advances. The player slides nowhere but the walk animation plays.

4. **No walk frame synchronisation across direction changes.** When the player switches direction (e.g., south → east), `animTime` continues accumulating from wherever it was. The walk cycle continues from mid-stride on the new direction row. This can look like a frame-skip stutter.

5. **Frame timing at 150ms per frame.** This is borderline too slow for an 8-frame walk at 64px scale. At 150ms/frame × 8 frames = 1200ms per full cycle — roughly 0.83 steps/second. Standard comfortable walk speed for a humanoid sprite is 1.5–2 steps/second (80–120ms/frame). Industry references (Slynyrd, finalbossblues) recommend 8-frame cycles at ~100ms/frame for natural walking.

**Primary recommendation:** Fix the five issues above in one focused plan. No new libraries needed. All changes are in `PlayerInput.ts`, `GameState.ts`, `spriteStates.ts`, and `OfficePage.tsx`.

## Architecture Patterns

### Pattern 1: Animation State Machine (idle → walk → idle)

The professional approach for top-down 2D sprite animation is a finite state machine with two states: `IDLE` and `WALK`. Transitions are:

```
IDLE → WALK: when isMoving becomes true AND actual displacement dx+dy is non-zero
WALK → IDLE: when isMoving becomes false
```

In this codebase the "state machine" is implicit: `animTime > 0` means WALK, `animTime === 0` means IDLE. This is fine — no need to introduce a separate enum. The fix is ensuring the transition guard uses actual displacement, not key booleans.

**Correct isMoving guard:**

```ts
// Current (wrong — fires even if W+S cancel out):
const isMoving = up || down || left || right

// Correct (fires only when actual movement occurred):
const isMoving = dx !== 0 || dy !== 0
// Note: use the NORMALISED dx/dy (after diagonal scaling) for maximum accuracy,
// but the pre-normalised raw dx/dy is also correct since dx===0 iff left===right
```

The distinction matters for the edge case of opposing keys (W+S or A+D simultaneously). Using `dx !== 0 || dy !== 0` is correct because if both W and S are held, `dy = 1 - 1 = 0`, so `isMoving = false`.

### Pattern 2: Walk Frame Timing — 100ms per frame

Industry standard for 8-frame pixel art walk cycles at 64px:

| Source | Recommendation | Notes |
|--------|---------------|-------|
| Slynyrd (2024) | 8 frames at natural walking pace | Converts from 60fps reference footage |
| finalbossblues | 8 frames = optimal for pixel art | "Never need more than 8" |
| peerdh.com | 100ms per frame (10fps walk anim) | Explicit code example |
| Phase 16.2 | 150ms per frame | Current value — too slow |

**Recommended constant:**
```ts
export const WALK_FRAME_DURATION_MS = 100  // was 150 — 10fps walk cycle, natural humanoid gait
```

At 100ms × 8 frames = 800ms/cycle = 1.25 full cycles/second. At PLAYER_SPEED = 120 px/s and a 64px sprite, the player travels ~96px per stride — visually coherent with the foot movement.

### Pattern 3: Direction-aware idle frame

When the player stops, the sprite must show the correct direction's idle frame 0 — not a fixed row.

The current code already does this correctly in principle: `DIRECTION_ROWS[gameState.player.direction]` is used for the row, and `direction` is only updated when `dx !== 0 || dy !== 0` (see `deriveDirection`). So the last direction is preserved when idle.

**This is already correct.** No change needed here — just verify it works once the walk row fix is in place.

### Pattern 4: Direction change does not reset animTime

The current implementation does NOT reset `animTime` on direction change. This is the correct behaviour:

- Resetting animTime on direction change causes a visible stutter: the animation snaps back to frame 0 on every direction input, which feels worse than continuing the cycle mid-stride.
- Professional games (e.g., classic top-down RPGs) do NOT reset the cycle on direction change — they let it continue.

**No change needed here.** The current code handles this correctly.

### Pattern 5: Row selection at render site

After Phase 16.4 lands, the render site must use `STATE_ROW_OFFSET.walk` (= 32) for moving, and `STATE_ROW_OFFSET.idle` (= 0) for stopped:

```ts
// OfficePage.tsx line 115 — current (wrong):
const stateOffset = gameState.player.animTime > 0 ? STATE_ROW_OFFSET.blocked : STATE_ROW_OFFSET.idle

// Correct (after Phase 16.4 adds STATE_ROW_OFFSET.walk):
```

If Phase 16.4 is incomplete when Phase 16.5 executes, `STATE_ROW_OFFSET.walk` may not exist yet. Phase 16.5 must either depend on 16.4 being done OR define the constant itself.

### Anti-Patterns to Avoid

- **Resetting animTime on direction change:** Creates visible frame-0 stutter on every direction input. Do not add this.
- **Using key booleans for isMoving instead of dx/dy:** Allows opposing keys to play walk animation while standing still ("moonwalking").
- **Speed-coupled animation:** Tying WALK_FRAME_DURATION_MS to PLAYER_SPEED makes the animation feel mechanical and breaks when speed changes. Keep them independent.
- **Interpolating between sprite frames:** Pixel art looks correct with crisp frame snaps. Never alpha-blend between frames.
- **Per-direction animTime:** Adds state complexity with no visual benefit. One accumulator for all directions is standard.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Frame easing / tweening | Interpolation between adjacent walk frames | Direct integer frame snap | Pixel art is intentionally discrete; blending destroys the aesthetic |
| Animation scheduler | Custom timer/setTimeout per animation | `Math.floor(animTime / DURATION_MS) % COUNT` | One-liner; zero overhead; frame-rate independent |
| Separate walk speed from player speed | Velocity-derived frame timing | Independent `WALK_FRAME_DURATION_MS` constant | Decoupled constants are easier to tune and test |

## Common Pitfalls

### Pitfall 1: `isMoving` uses key booleans, not displacement

**What goes wrong:** Player holds W+S simultaneously. `up=true`, `down=true`. `dy = 0`. Player doesn't move, but `animTime` accumulates, playing the walk animation while the character is stationary. This is the "moonwalking" bug.

**Why it happens:** `const isMoving = up || down || left || right` — this fires when any key is down regardless of whether they cancel.

**How to avoid:** Use `const isMoving = dx !== 0 || dy !== 0` after computing `dx`/`dy`. The raw (pre-normalised) dx/dy is sufficient since `dx = (right?1:0) - (left?1:0)` — it's zero when both or neither are pressed.

**Warning signs:** Character plays walk animation with no visible movement on screen.

### Pitfall 2: STATE_ROW_OFFSET.walk not defined (Phase 16.4 incomplete)

**What goes wrong:** Phase 16.5 references `STATE_ROW_OFFSET.walk` but Phase 16.4 hasn't added it to `spriteStates.ts`. TypeScript compilation error.

**How to avoid:** Phase 16.5 plan must either (a) depend on Phase 16.4 being complete and verified, or (b) add `walk: 32` to `STATE_ROW_OFFSET` as part of its own work.

**Warning signs:** `Property 'walk' does not exist on type 'Record<AnimationState, number>'` TypeScript error.

### Pitfall 3: Frame timing too slow makes animation feel floaty

**What goes wrong:** At 150ms/frame, the 8-frame walk cycle takes 1.2 seconds per full step. For a 64px sprite at 120 px/s, this means the sprite visually travels 144px per stride — almost 2.5 sprite-widths per foot-plant. Legs appear to slide rather than walk.

**Why it happens:** 150ms was chosen in Phase 16.2 based on a 4-frame cycle (Phase 16.2 originally specified WALK_FRAME_COUNT=4, then changed to 8 without adjusting timing).

**How to avoid:** Change `WALK_FRAME_DURATION_MS` from 150 to 100. This gives 800ms/cycle at 8 frames, matching the 60fps reference footage rate used in professional pixel art tutorials.

**Warning signs:** Walk looks like sliding, moonwalking, or "ghost sliding on ice."

### Pitfall 4: WALK_FRAME_COUNT test assertion uses wrong expected value

**What goes wrong:** `PlayerInput.test.ts` has test: "col formula wraps from frame 3 back to frame 0" — this test name is misleading (noted in VERIFICATION.md) because WALK_FRAME_COUNT is 8, not 4. The test passes but tests the wrong wrap boundary.

**How to avoid:** When updating WALK_FRAME_DURATION_MS, also review the wrap test: ensure it asserts `% 8 = 0` at frame 8 (animTime = 800ms), not `% 4`.

**Warning signs:** Test named "wraps at frame 3" but count is 8.

### Pitfall 5: Idle direction mismatch after teleport

**What goes wrong:** Player teleports to an NPC via click. `gameState.player.direction` remains `'south'` (the default). The teleport sets `x`/`y` but doesn't update `direction`. If the NPC is to the north, the player sprite faces south even after teleporting north.

**Why it happens:** The teleport in OfficePage click handler sets `x`/`y` only, not `direction`.

**How to avoid:** This is a pre-existing issue, not introduced by Phase 16.5. Flag it as a known limitation — out of scope unless the planner decides to include it.

## Code Examples

### Corrected isMoving guard in movePlayer()

```ts
// Source: verified PlayerInput.ts structure, fix derived from dx/dy computation already present

export function movePlayer(
  player: { x: number; y: number; direction: string; animTime: number },
  keys: ReadonlySet<string>,
  deltaMs: number,
): void {
  const dt = deltaMs / 1000
  const dist = PLAYER_SPEED * dt

  const up = keys.has('KeyW') || keys.has('ArrowUp')
  const down = keys.has('KeyS') || keys.has('ArrowDown')
  const left = keys.has('KeyA') || keys.has('ArrowLeft')
  const right = keys.has('KeyD') || keys.has('ArrowRight')

  let dx = (right ? 1 : 0) - (left ? 1 : 0)
  let dy = (down ? 1 : 0) - (up ? 1 : 0)

  if (dx !== 0 && dy !== 0) {
    dx *= INV_SQRT2
    dy *= INV_SQRT2
  }

  player.x = Math.max(0, Math.min(player.x + dx * dist, WORLD_W - 64))
  player.y = Math.max(0, Math.min(player.y + dy * dist, WORLD_H - 64))

  const direction = deriveDirection(dx, dy)
  if (direction !== null) {
    player.direction = direction
  }

  // FIX: use actual displacement, not key booleans
  const isMoving = dx !== 0 || dy !== 0  // was: up || down || left || right
  if (isMoving) {
    player.animTime += deltaMs
  } else {
    player.animTime = 0
  }
}
```
