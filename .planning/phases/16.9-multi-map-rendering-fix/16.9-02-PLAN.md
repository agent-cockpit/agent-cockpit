    3a. Update `buildObjectAlphaBounds` signature to accept `mapDir: string` as second parameter. Change the internal path from `` `/map/objects/${obj.filename}` `` to `` `${mapDir}/objects/${obj.filename}` ``.

    3b. Replace the single-map `Promise.all([...])` collision load block with a multi-map loop. The new logic should:
    1. Load the manifest from `/maps/maps-manifest.json` (one extra fetch before the loop)
    2. Load `tilemapRenderer` first (so `worldW/worldH` are available), or load in parallel and update bounds after
    3. For each entry in `manifest.maps` (in order):
       - Fetch `${entry.dir}/terrain-map.json` and `${entry.dir}/objects/manifest.json` in parallel
       - Call `buildObjectAlphaBounds(objects, entry.dir)` with the entry's dir
       - Call `collisionMap.loadTerrain(terrainData, { tileOriginX: entry.tileOriginX, tileOriginY: entry.tileOriginY, worldOriginX: entry.worldOriginX, worldOriginY: entry.worldOriginY, append: i > 0 })`
       - Call `collisionMap.loadObjects(objects, alphaBounds, { tileOriginX: entry.tileOriginX, tileOriginY: entry.tileOriginY, worldOriginX: entry.worldOriginX, worldOriginY: entry.worldOriginY, append: i > 0 })`
    4. After `tilemapRenderer.load()` resolves, update the mutable world bounds: `WORLD_W = tilemapRenderer.worldW; WORLD_H = tilemapRenderer.worldH` (import WORLD_W/WORLD_H as the mutable lets).

    3c. The `tilemapRenderer.load()` call in the engine setup useEffect should remain non-blocking (engine starts before assets are ready — this is correct). The manifest-based collision loop should share the same manifest data as the renderer to avoid double-fetching. Best approach: fetch the manifest once, pass it to both. Since `tilemapRenderer.load()` already fetches the manifest internally, the OfficePage collision loop can fetch it again independently (it's a tiny JSON, acceptable). Do NOT restructure the engine startup sequence.

    3d. After the loop completes, set:
    ```typescript
    // Expand world bounds to cover all loaded maps
    WORLD_W = tilemapRenderer.worldW || 3232  // fallback if renderer not done yet
    WORLD_H = tilemapRenderer.worldH || 3232
    ```
    The safest approach: chain the world bounds update inside `.then()` after `tilemapRenderer.load()`.

    **Note:** The existing `import { gameState, WORLD_W, WORLD_H } from '../game/GameState.js'` will continue to work because `let` exports in ES modules are live bindings — the imported names track the module variable. Reassigning `WORLD_W = tilemapRenderer.worldW` inside the module or from OfficePage via a setter function is NOT possible via the import binding. Use a setter: add `export function setWorldBounds(w: number, h: number) { WORLD_W = w; WORLD_H = h }` to GameState.ts, and call `setWorldBounds(tilemapRenderer.worldW, tilemapRenderer.worldH)` from OfficePage after load.

    Using ES module live bindings: export WORLD_W/WORLD_H as `let`. OfficePage imports them by name. After load, call `setWorldBounds(w, h)` which mutates the module-level variables. All other importers of WORLD_W/WORLD_H (Camera.ts, OfficePage.tsx click handler) will read the updated values on next access because ES module live bindings reflect mutations.
