# Phase 16.9: Multi-Map Rendering Fix - Research

**Researched:** 2026-04-13
**Domain:** Canvas 2D multi-image rendering, browser static-asset manifests, TypeScript game engine
**Confidence:** HIGH

## Summary

The current `TilemapRenderer` loads a single hardcoded `/map/map-composite.png` and has `WORLD_W/WORLD_H` baked into both the renderer (`WORLD_PX = 3232`) and `GameState.ts` (`WORLD_W = WORLD_H = 3232`). The fix is straightforward: introduce a static `maps-manifest.json` file under `/maps/` that lists every map's subdirectory name and pixel origin offset, load all composites in parallel at startup, and blit each one at its world-space offset each frame. Browsers cannot enumerate directories, so a manifest is the only viable approach.

Investigation of the actual `public/` tree confirms there is currently only ONE map (`/map/map-composite.png`). A second map does not yet exist on disk — the user wants to add a second export and have the system handle it. The fix therefore also needs to define and document the directory convention (`/maps/map1/`, `/maps/map2/`) and manifest format so the user can drop new map exports in and they will appear automatically.

`CollisionMap` hardcodes `TILE_ORIGIN_X = -46`, `TILE_ORIGIN_Y = -43` and `OBJECT_ORIGIN_X/Y` from those constants. When a second map has a different `boundingBox.minX/minY`, these values will be wrong for that map's tiles and objects. The fix must pass per-map origin offsets into `CollisionMap` rather than relying on module-level constants.

**Primary recommendation:** Introduce `/maps/maps-manifest.json`, migrate the existing map to `/maps/map1/`, create `TilemapRenderer` to accept a manifest entry array, and extend `CollisionMap.loadTerrain()`/`loadObjects()` to accept a per-map world origin offset so tile and object coordinates are correct for each map.

### Pattern 1: Static Manifest File

**What:** A single `maps-manifest.json` file lists every map. Browser cannot enumerate directories, so this is the only viable auto-discovery mechanism.

**When to use:** Any time new maps are added — the user drops files in the correct subdirectory and updates the manifest.

**Example manifest schema:**
```typescript
// /maps/maps-manifest.json
interface MapManifestEntry {
  id: string           // e.g. "map1" — used as a stable key
  dir: string          // URL path prefix, e.g. "/maps/map1"
  worldOriginX: number // pixel X where this map's (0,0) sits in world space
  worldOriginY: number // pixel Y where this map's (0,0) sits in world space
  widthPx: number      // composite image width (used to expand WORLD_W)
  heightPx: number     // composite image height (used to expand WORLD_H)
}

interface MapsManifest {
  version: string
  maps: MapManifestEntry[]
}
```

**Concrete example for the current single map:**
```json
{
  "version": "1.0.0",
  "maps": [
    {
      "id": "map1",
      "dir": "/map",
      "worldOriginX": 0,
      "worldOriginY": 0,
      "widthPx": 3232,
      "heightPx": 3232
    }
  ]
}
```

### Pattern 2: Multi-Image TilemapRenderer

**What:** `TilemapRenderer` holds an array of loaded `{ img, entry }` records instead of a single `mapImg`. `blit()` iterates over all records and draws each at `(-camX + entry.worldOriginX, -camY + entry.worldOriginY)`.

**When to use:** Every render frame (inside `ctx.scale(zoom, zoom)` block).

```typescript
// Source: Canvas 2D API, official MDN spec
interface LoadedMap {
  img: HTMLImageElement
  entry: MapManifestEntry
}

export class TilemapRenderer {
  private maps: LoadedMap[] = []
  private _ready = false

  get ready(): boolean { return this._ready }

  // worldW/worldH computed as union of all map rectangles
  worldW = 0
  worldH = 0

  async load(manifestUrl = '/maps/maps-manifest.json'): Promise<void> {
    const manifest: MapsManifest = await fetch(manifestUrl).then(r => r.json())
    const loaded = await Promise.all(
      manifest.maps.map(entry =>
        loadImage(`${entry.dir}/map-composite.png`).then(img => ({ img, entry }))
      )
    )
    this.maps = loaded
    // World bounds = union of all map rectangles
    this.worldW = Math.max(...loaded.map(m => m.entry.worldOriginX + m.entry.widthPx))
    this.worldH = Math.max(...loaded.map(m => m.entry.worldOriginY + m.entry.heightPx))
    this._ready = true
  }

  blit(ctx: CanvasRenderingContext2D, camX: number, camY: number): void {
    if (!this._ready) return
    for (const { img, entry } of this.maps) {
      ctx.drawImage(img, entry.worldOriginX - camX, entry.worldOriginY - camY)
    }
  }
}
```

### Pattern 3: Per-Map Origin in CollisionMap

**What:** `CollisionMap.loadTerrain()` and `loadObjects()` currently hardcode `TILE_ORIGIN_X = -46` and `TILE_ORIGIN_Y = -43` as module-level constants derived from map1's `boundingBox.minX/minY`. For a second map with different tile origins this will produce wrong world pixel coords.

**Solution:** Add a `worldOffset` parameter (`worldOriginX`, `worldOriginY`) to both load methods. The tile-to-pixel conversion adds this offset so every map's solid tiles land at the correct absolute world pixel position.

```typescript
// Updated signature
loadTerrain(
  data: TerrainMapData,
  opts: { tileOriginX: number; tileOriginY: number; worldOriginX?: number; worldOriginY?: number }
): void

loadObjects(
  objects: ObjectEntry[],
  alphaBounds: Record<string, ObjectAlphaBounds>,
  opts: { tileOriginX: number; tileOriginY: number; worldOriginX?: number; worldOriginY?: number }
): void
```

For the existing map: `tileOriginX = -46, tileOriginY = -43, worldOriginX = 0, worldOriginY = 0` — producing identical output to current code. No behaviour change for the existing map, but now correct for any second map.

### Pattern 4: Propagate worldW/worldH from Renderer to GameState

**What:** `WORLD_W` and `WORLD_H` in `GameState.ts` are currently `const` exports (`3232`). After multi-map load they must reflect the union of all loaded maps. `updateCamera` and click-to-teleport both use these values for bounds clamping.

**Solutions (in order of simplicity):**
1. Make `WORLD_W`/`WORLD_H` mutable module-level `let` exports — updated after `tilemapRenderer.load()` resolves.
2. Pass the renderer's `worldW`/`worldH` directly into `updateCamera()` and the bounds clamp expressions in `OfficePage.tsx`.

Option 2 is cleaner (no mutable module state) but requires threading the values through. Option 1 is smaller diff. Either works — the planner should choose one and document it.

### Pattern 5: Multi-Map CollisionMap Loading in OfficePage

**What:** `OfficePage.tsx` currently calls one `fetch('/map/terrain-map.json')` and one `fetch('/map/objects/manifest.json')`. With multiple maps, it must iterate the manifest entries and call `collisionMap.loadTerrain()` / `collisionMap.loadObjects()` once per map, each with that map's origin offsets.

```typescript
// Pseudocode — after tilemapRenderer.load() resolves
const mapsManifest = await fetch('/maps/maps-manifest.json').then(r => r.json())
for (const mapEntry of mapsManifest.maps) {
  const [terrain, objManifest] = await Promise.all([
    fetch(`${mapEntry.dir}/terrain-map.json`).then(r => r.json()),
    fetch(`${mapEntry.dir}/objects/manifest.json`).then(r => r.json()),
  ])
  const tileOriginX = mapEntry.tileOriginX   // from manifest or from map.json boundingBox.minX
  const tileOriginY = mapEntry.tileOriginY
  const alphaBounds = await buildObjectAlphaBounds(objManifest.objects, mapEntry.dir)
  collisionMap.loadTerrain(terrain, { tileOriginX, tileOriginY, worldOriginX: mapEntry.worldOriginX, worldOriginY: mapEntry.worldOriginY })
  collisionMap.loadObjects(objManifest.objects, alphaBounds, { tileOriginX, tileOriginY, worldOriginX: mapEntry.worldOriginX, worldOriginY: mapEntry.worldOriginY })
}
```

> `tileOriginX/Y` can be embedded in the manifest or read from each map's `map.json` at load time. Embedding in the manifest avoids an extra fetch.

### Anti-Patterns to Avoid

- **Hard-coded image count:** Never write `load('/maps/map1/...', '/maps/map2/...')` as separate method calls — add more maps to the manifest only.
- **Sequential image loading:** Never `await loadImage(a); await loadImage(b)` — use `Promise.all` to load concurrently.
- **Re-using single `mapImg` field:** Replacing `this.mapImg` per iteration was the original bug (last-write-wins). Must use an array.
- **Global `addTile` accumulation without `loadTerrain` reset:** Current `loadTerrain` calls `this.solidTiles.clear()`. With multiple maps this clear must NOT happen between maps — either load all maps before the first clear, or change the strategy to accumulate (append) rather than replace on each `loadTerrain` call.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Directory listing in browser | Custom server endpoint that lists `/maps/` | Static `maps-manifest.json` — browsers cannot read directories |
| World bounds union | Manual bounding box math | `Math.max(...maps.map(m => m.entry.worldOriginX + m.entry.widthPx))` |
| Parallel asset loading | Custom promise queue | `Promise.all()` on the manifest entries array |

**Key insight:** Static manifests are the correct primitive for "what assets exist" in a Vite/browser environment. No runtime directory scanning is possible or needed.

## Common Pitfalls

### Pitfall 1: CollisionMap.loadTerrain() Clears on Each Call
**What goes wrong:** `loadTerrain` starts with `this.solidTiles.clear()`. If called once per map in a loop, the second call clears the first map's solid tiles before inserting its own — only the last map's terrain survives.
**Why it happens:** Original design assumed one map per lifetime.
**How to avoid:** Change `loadTerrain` to accept an `append = false` flag, OR load all maps' terrain data first then do a single batch insert, OR call `loadTerrain` in "append mode" by removing the `clear()` when appending.
**Warning signs:** Player walks through walls of the first map after second map loads.

### Pitfall 2: Object Origin Constants are Hardcoded for Map1
**What goes wrong:** `OBJECT_ORIGIN_X = (1 - TILE_ORIGIN_X) * TILE_SIZE = 1504` and `OBJECT_ORIGIN_Y = (2 - TILE_ORIGIN_Y) * TILE_SIZE = 1440` are computed from map1's `boundingBox`. A second map has different values.
**Why it happens:** Module-level constants computed once at file load.
**How to avoid:** Pass `tileOriginX/Y` and `worldOriginX/Y` as parameters to `loadObjects()`. Compute `objectOriginX/Y` inside the method using the passed values.

### Pitfall 3: WORLD_W / WORLD_H Bounds Clamp Stays at 3232
**What goes wrong:** `updateCamera` and click-to-teleport bounds use `WORLD_W`/`WORLD_H`. If these stay at 3232, the camera can't reach a second map placed beyond that boundary.
**Why it happens:** Currently `const` module exports in `GameState.ts`.
**How to avoid:** After `tilemapRenderer.load()` resolves, update world bounds before the engine's first `update()` tick. Use `tilemapRenderer.worldW` and `tilemapRenderer.worldH` everywhere bounds are needed.

### Pitfall 4: Object PNG Paths Constructed with Wrong Base Dir
**What goes wrong:** `buildObjectAlphaBounds` in `OfficePage.tsx` currently constructs paths as `/map/objects/${obj.filename}`. With multiple maps, objects from map2 live under `/maps/map2/objects/`.
**Why it happens:** Hardcoded `/map/objects/` prefix.
**How to avoid:** Pass the map's `dir` base path to `buildObjectAlphaBounds` so it becomes `` `${mapDir}/objects/${obj.filename}` ``.

### Pitfall 5: Concurrent Loads Then Sequential Collision Init
**What goes wrong:** `tilemapRenderer.load()` fires in one `Promise` and collision data fires in a separate `Promise.all` in `OfficePage.tsx`. If these are decoupled, the collision data for map2 may try to load before `maps-manifest.json` is fetched.
**Why it happens:** Structural separation of renderer load and collision load.
**How to avoid:** Share the manifest fetch between both operations, or sequence: fetch manifest → `tilemapRenderer.load(manifest)` → `collisionMap` loads (using same manifest entries).

### CollisionMap per-map origin injection
```typescript
loadTerrain(data: TerrainMapData, opts: {
  tileOriginX: number
  tileOriginY: number
  worldOriginX?: number
  worldOriginY?: number
  append?: boolean
}): void {
  if (!opts.append) this.solidTiles.clear()
  const { tileOriginX, tileOriginY, worldOriginX = 0, worldOriginY = 0 } = opts
  for (const cell of data.cells) {
    if (SOLID_TERRAIN_IDS.has(cell.terrainId)) {
      const px = (cell.x - tileOriginX) * TILE_SIZE + worldOriginX
      const py = (cell.y - tileOriginY) * TILE_SIZE + worldOriginY
      const tx = Math.floor(px / TILE_SIZE)
      const ty = Math.floor(py / TILE_SIZE)
      this.solidTiles.add(`${tx},${ty}`)
    }
  }
}
```

## Open Questions

1. **Should existing `/map/` directory be migrated to `/maps/map1/`?**
   - What we know: All current references hardcode `/map/`. Moving it is a find-and-replace across 4 files.
   - What's unclear: User preference — keep `/map/` as map1's path (manifest just points to it) vs full migration.
   - Recommendation: Keep `/map/` for map1 initially. Only create `/maps/` directory for the manifest file + future maps. The manifest `dir` for map1 can be `/map`. Zero migration risk.

2. **Where does `tileOriginX/Y` for each map come from in the manifest?**
   - What we know: It is `mapConfig.boundingBox.minX/minY` from each map's `map.json`. For map1: `(-46, -43)`.
   - What's unclear: Should the manifest embed these values (faster, one-source) or should the renderer re-fetch each `map.json` at startup?
   - Recommendation: Embed in manifest — avoids extra fetches and keeps the manifest as the single config source. When user exports a new map they add one entry to the manifest.

3. **CollisionMap `solidTiles.clear()` strategy with multiple loads**
   - What we know: Current design clears on each `loadTerrain` call. Multi-map requires accumulation.
   - Recommendation: Add `append?: boolean` parameter. Default `false` for backward compat (tests pass unchanged). Pass `append: true` for all maps after the first.
