## Character Creation

### Parameters Reference Table

| Parameter | Type | Default | Valid Values | Notes |
|---|---|---|---|---|
| `description` | string (required) | ‚Äî | 1‚Äì2000 chars | Character appearance description |

| Preset | Description |
|---|---|

### Cost per Character Creation

| Mode | Directions | Cost (generations) | Notes |
|---|---|---|---|
| `standard` | 4 | **1 generation** | Template-based skeleton |
| `standard` | 8 | **1 generation** | Template-based skeleton |
| `pro` | 8 (forced) | **20‚Äì40 generations** | Depends on character `size` |

**Key insight:** Standard mode always costs 1 generation regardless of direction count. Pro mode is 20‚Äì40x more expensive and does not support custom style parameters (outline/shading/detail/proportions ignored).

### Async vs Sync Behavior

- **Always async** ‚Äî creation returns immediately with `background_job_id` and `character_id`
- Processing takes 2‚Äì3 minutes for 4 directions, 3‚Äì5 minutes for 8 directions or pro mode
- Poll with `get_character(character_id)` to check completion

**Response on creation:**
```json
{
  "background_job_id": "uuid",
  "character_id": "uuid",
  "status": "processing"
}
```

The `character_id` is available immediately even before generation completes ‚Äî animations can be queued before the base character finishes.

### Output Format and Download

**Per `get_character` response:**

- `rotation_urls`: Object with URLs per direction (`south`, `west`, `east`, `north`, plus diagonals for 8-dir)
  - Each URL is an individual PNG for that direction's idle pose
  - Format: public URL, no auth required (UUID acts as access key)
- `zip_url`: `/mcp/characters/{id}/download` ‚Äî ZIP of all rotations + animations
- `preview_url`: Public URL to south-facing direction sprite
- `animations[]`: List of animation objects with status + frame URLs
- `style_settings`: Object recording the outline/shading/detail used
- `size`: `{width: N, height: N}` ‚Äî canvas size (~40% larger than requested character size)

**Frame format:**
- Individual PNG files per direction per animation frame
- No sprite sheet generated by default ‚Äî individual frames must be stitched by the consumer
- Frame dimensions are consistent across all frames (same size as character `size` canvas)

**Metadata returned by `get_character`:**
```json
{
  "id": "uuid",
  "name": "string",
  "prompt": "original description",
  "size": {"width": 68, "height": 68},
  "directions": 4,
  "created_at": "ISO timestamp",
  "animation_count": 3,
  "template_id": "mannequin",
  "view": "low top-down",
  "rotation_urls": {
    "south": "https://...",
    "west": "https://...",
    "east": "https://...",
    "north": "https://..."
  },
  "style_settings": {"outline": "...", "shading": "...", "detail": "..."},
  "animations": [
    {
      "name": "walk",
      "status": "completed",
      "frames": {"south": ["url1", "url2", ...], ...}
    }
  ]
}
```

### Frame Count per Animation

Frame counts are **determined by the template** and cannot be configured in template mode. From the API spec:

- Template mode: "Typically 4‚Äì6 frames for most animations" (spec quote)
- Confirmed specific counts:
  - `fight-stance-idle-8-frames`: **8 frames**
  - `running-4-frames`: **4 frames**
  - `running-6-frames`: **6 frames**
  - `running-8-frames`: **8 frames**
  - `walking-4-frames`: **4 frames**
  - `walking-6-frames`: **6 frames**
  - `walking-8-frames`: **8 frames**
- General estimate: **4‚Äì8 frames** per animation template

In v3 (custom) mode, frame count is configurable: 4‚Äì16 frames (must be even), default 8.

### Cost per Animation

| Mode | Cost per direction | Notes |
|---|---|---|
| `template` | **1 generation/direction** | Uses skeleton template; consistent results |
| `v3` (custom, no template) | **20‚Äì40 gens/direction** | AI-generated frames from text |
| `pro` (custom sequential) | **20‚Äì40 gens/direction** | Uses completed directions as reference |

**Important:** For a 4-direction character with template animations:
- Each animation call = 4 √ó 1 = **4 generations** total

For an 8-direction character:
- Each animation call = 8 √ó 1 = **8 generations** total

### Relationship: Character vs Animation

- Animations build **on top of** an existing character (they inherit its `character_id`)
- Character must be created first, but animation jobs can be **queued immediately** (before character finishes processing)
- Animations are stored with the character and accessible via `get_character`
- Multiple animations per character are stored and accessible independently

### `directions` Parameter in Animation Context

- `directions`: list of strings like `["south", "east", "north", "west"]`
- **Template mode default:** all directions the character was created with
- **Custom mode default:** south only
- Valid values: `south`, `north`, `east`, `west`, `south-east`, `south-west`, `north-east`, `north-west`

## `get_character` Output Details

### What is returned

- `rotation_urls`: per-direction static pose URLs (individual PNGs, not sprite sheet)
- `animations[]`: array of animation objects, each with:
  - `name`: animation name
  - `status`: `processing`, `completed`, `failed`
  - `frames`: object keyed by direction, each value is `string[]` of frame URLs
- `zip_url`: download link for ZIP of all assets
- `preview_url` (when `include_preview=true`): composite preview image showing all rotations
- Pending jobs: animations still processing are listed with job IDs

### Download method

```
GET /mcp/characters/{character_id}/download
```
Returns a ZIP file. No authentication required ‚Äî UUID functions as the access key.

Individual frame URLs are public CDN links returned in the `frames` dict.

## Practical Constraints

### Size Recommendations

- **Minimum viable:** 16px (supported but produces low-quality results at this size)
- **Our target:** 32px character size ‚Äî canvas output will be ~`32 √ó 1.4 = ~45px`
- **Recommended minimum (per docs):** Sizes above 24px produce "better quality" (isometric tile doc quote, consistent with character quality)
- **Maximum for our use case:** 48px ‚Äî standard default, good quality, 40% larger canvas (~68px)
- **Note on sprite sheet stitching:** We must stitch individual frame PNGs ourselves. No sprite sheet is generated automatically.

### Quality vs Cost Tradeoffs

| Quality Level | outline | shading | detail | Relative Cost | Notes |
|---|---|---|---|---|---|
| Budget | `single color black outline` | `flat shading` | `low detail` | 1 gen | Acceptable for small sizes |
| Standard | `single color black outline` | `basic shading` | `medium detail` | 1 gen | Default settings, good for 32px |
| High | `single color outline` | `medium shading` | `medium detail` | 1 gen | Better at 48px+ |
| Premium | `selective outline` | `detailed shading` | `highly detailed` | 1 gen | Best at 64px+ |

**Key insight:** Quality parameters do NOT affect generation cost in standard mode ‚Äî all quality levels cost 1 generation. Only `pro` mode changes cost. Choose higher quality settings at no additional cost.

### API Limits

From the spec:
- **Rate limiting:** `429 Too Many Requests` returned when exceeded
- **Concurrent jobs:** `429` also covers "too many concurrent jobs"
- **Max size:** 128px per character
- **Max frame count (v3 mode):** 16 frames
- **Subscription type:** `generations` quota (our budget: 2000 total)
- **Processing time:** 2‚Äì5 minutes per character/animation job

### Sprite Sheet Assembly Strategy

Since the API returns individual frame PNGs (not a sprite sheet), we need a post-processing step:

1. Fetch all frame URLs from `get_character` response
2. Download each PNG
3. Stitch horizontally (or in grid) using Node.js canvas / Sharp / jimp
4. Save as `/sprites/agent-sheet.png`

**Recommended sheet layout:**
- One row per `AgentAnimState` (8 rows)
- Each row contains frames for the animation
- All frames 32px √ó 32px (or the canvas size returned by the API)
- CSS background-position selects the correct row and frame

## Recommended Approach for Agent Cockpit

### Optimal Parameters for a 32px Space-Themed Agent

```
create_character(
  description: "sci-fi space station worker, small humanoid robot with glowing visor, 
                compact body, clean sci-fi aesthetic, pixel art style",
  body_type: "humanoid",
  mode: "standard",           // 1 generation cost
  n_directions: 4,            // S/W/E/N ‚Äî sufficient for top-down view
  size: 32,                   // matches our existing spec
  proportions: '{"type": "preset", "name": "chibi"}',  // large head = readable at 32px
  outline: "single color black outline",  // clean retro pixel art look
  shading: "basic shading",              // sufficient for 32px
  detail: "medium detail",               // balanced
  ai_freedom: 600,            // slightly constrained for consistency
  view: "low top-down"        // matches our game view
)
```

**Why chibi proportions:** At 32px, a large head ratio is critical for facial readability. Chibi style is standard for top-down RPG characters and ensures the glowing visor is visible.

**Why 4 directions:** Our current `AgentSprite` does not use directional rendering ‚Äî the sprite faces one direction and plays state animations. 4 directions gives us flexibility if we add walking direction later at half the cost of 8.

### Alternative: Skip Animation Templates, Use v3 Custom

If none of the 47 templates map well to our states:
- Use `v3` mode with `action_description` like "robot standing still, breathing gently"
- Cost: **20‚Äì40 gen/direction** √ó 4 directions = **80‚Äì160 gen per animation**
- For 8 animations: **640‚Äì1280 generations** ‚Äî still within 2000 budget but leaves less margin
- **Recommendation:** Try templates first; fall back to v3 for states with no good template match

### Directional Strategy

For the current Office Mode implementation, only the **south direction** (character facing the viewer) is needed for the sprite sheet. We can:

1. Generate with `n_directions: 4`
2. Use only the `south` direction frames for the sprite sheet
3. This halves the usable frames per animation but saves animation time and keeps options open

OR:
1. Generate with `n_directions: 4`, animate with `directions: ["south"]` only
2. Saves 75% of animation generation cost
3. **Recommended if we only need one facing direction**

With south-only animation at 4-direction character:
- 8 animations √ó 1 direction √ó 1 gen = **8 generations total** for all animations
- **Extremely budget-efficient**

## Key Decisions for Plan 03 (Budget Planning)

1. **Standard mode is always 1 gen per character** ‚Äî no reason to ever use `pro` for our use case
2. **Template animations are always 1 gen/direction** ‚Äî always prefer templates over custom animations
3. **South-only animation costs 8 total gens for all 8 states** ‚Äî dramatically cheaper than all-directions
4. **Sprite sheet must be assembled manually** ‚Äî requires post-processing step in the generation pipeline
5. **Size 32px maps to ~45px canvas output** ‚Äî the CSS sprite renderer needs to account for the larger canvas, not 32px exactly, OR we crop the output
6. **`breathing-idle`** is the best `waiting` template. **`falling-back-death`** is the best `failed`. **`jumping-1`** is the best `completed`
7. **Budget risk is near-zero** ‚Äî even with 3 retries and all 8 directions, total cost ‚âà 39 generations (2% of 2000 budget)
