### Strategy: South-Only Animation (Recommended)

The current `AgentSprite` implementation renders the sprite facing one direction (south = facing viewer) and switches animation rows based on `AgentAnimState`. Multi-direction rendering is not implemented. Therefore:

- Generate character with `n_directions: 4` (keeps future directional options open)
- Animate with `directions: ["south"]` only
- **This reduces 8-state animation cost from 32 gen (4-dir) to 8 gen (south-only)**
- If directional walk is added in a future phase, remaining directions can be animated then

### Prompt Adjustment on Retry

When a generation produces wrong output, adjust in this order:

1. **Description first:** Rephrase the character description to be more explicit about pose/action. Add: "pixel art, hard outline, flat shading, NO anti-aliasing" if output looks non-pixel.
2. **ai_freedom adjustment:** Lower `ai_freedom` (toward 100) for more template-constrained results. Raise toward 999 if the result is too generic.
3. **Size adjustment:** Try `size: 48` instead of 32 — larger canvas gives the model more room to work, then scale down in post-processing.
4. **Anchor prompt addition:** Prepend "FTL crew member style, Stardew Valley proportions" to anchor against known reference aesthetics.

### If Tier 1 Exceeds Budget Projection

The nominal Tier 1 budget is 23 images. If retries push it above 100 images:

1. Cap at 100 images for Tier 1. Stop animating and accept best available results.
2. Defer Tier 2 to a second session.
3. Document which states are animation-pending in `GENERATION-LOG.md`.
4. Note: even 100 images for Tier 1 leaves 1700 images for Tier 2 + 3 + reserve. This scenario is essentially impossible to actually exhaust.

### Sprite Sheet Assembly Requirements

The PixelLab API returns **individual frame PNGs** — not a sprite sheet. After generating all state animations:

1. **Download all frames** from `get_character` response `frames` field (keyed by direction → `["url1", "url2", ...]`).
2. **Verify dimensions:** Each frame is ~45×45px canvas when `size: 32` is used (canvas = size × 1.4). The sprite renderer may need CSS adjustment from 32px to 45px, OR crop each frame to 32×32 center.
3. **Stitch frames into rows:**
   - Row 0: `planning` state frames (left to right)
   - Row 1: `coding` state frames
   - Row 2: `reading` state frames
   - Row 3: `testing` state frames
   - Row 4: `waiting` state frames
   - Row 5: `blocked` state frames
   - Row 6: `completed` state frames
   - Row 7: `failed` state frames
4. **Sheet dimensions:** 8 rows × (max_frames_per_state × frame_width). Frame counts vary by template (4–8 frames). Use padding to make all rows the same width.
5. **Output format:** 32-bit PNG (RGBA) with alpha transparency preserved.
6. **Tool recommendation:** Node.js `sharp` library (`pnpm add sharp`) supports PNG compositing without canvas API. See Plan 04 for implementation details.
