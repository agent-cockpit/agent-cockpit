**Purpose:** Curated reference library for the pixel art redesign of Agent Mission Control. Covers game references, PixelLab output characteristics, color palette options, state communication strategies, and prompt templates.

**Scope:** This document feeds Plan 04 (design consultation) and the generation phase.

## Game References

### 1. FTL: Faster Than Light (Subset Games, 2012)

**Why it's relevant:** Direct thematic match. FTL is a space station crew management game where small sprites represent individual crew members performing specific tasks (piloting, engineering, fighting, dying). The visual grammar for "crew in different functional states" was largely invented here.

**Character size:** 20×26px base sprite, displayed at 2x scale (40×52px on screen).

**Color palette approach:**
- Strict limited palette: 4–6 colors per crew member + 1–2 tones for outfit variation.
- Dark ship interior backgrounds (near-black: `#0d0d1a`, `#111122`) with bright sprite fills.
- Crew color-coded by race/species, not by job. Job state communicated through animation, not color.

**Idle vs action state differentiation:**
- Idle: 2-frame loop, subtle chest-breathing animation. Character stands upright.
- Running: 4-frame walk cycle, sprite leans slightly forward.
- Working at station: Arm movement animation (2–3 frames), body stays in place.
- Dying: Explicit "fall" animation, screen flash effect.

**What "blocked/thinking/working" looks like:**
- Working = stationed at a console, arm-raise loop (2 frames).
- Blocked = crew cannot pathfind to destination. Visual: crew stands still at old position. No dedicated "blocked" expression — purely positional.
- No "thinking" state exists in FTL. Closest: crew idle at a room with no task assigned.

**Outline style:** Hard 1px black outline on all sprite edges. No anti-aliasing. Sub-pixel detail handled by dithering inside the outline.

**Key takeaways for Agent Mission Control:**
- Hard outlines work at 32px. They preserve readability even on complex backgrounds.
- State should be communicated through animation type (static vs moving vs arm-raise loop), not just color.
- Small sprites at 32px can be expressive if the silhouette changes between states. FTL crew have a distinct silhouette per action that reads even at thumbnail size.
- Our 8 states need visual silhouette variation, not just animation — the "shape" of blocked should differ from coding.

**URLs:** https://store.steampowered.com/app/212680/FTL_Faster_Than_Light/ | Sprite sheets: documented extensively on The Spriters Resource

### 2. Celeste (Maddy Makes Games, 2018)

**Why it's relevant:** Celeste's protagonist Madeline is 8×8px in the base sprite sheet but contains enormous expressiveness through color, outline, and frame count. It is the gold standard for "maximum expression, minimum pixels."

**Character size:** 8×8px sprite, displayed at 6x (48×48px on screen effective size).

**Color palette approach:**
- Very limited: 3–4 colors per character state (highlight, midtone, shadow, outline).
- Color shifts dramatically per state: Madeline turns red/dark when corrupted, blue-white at crystal heart.
- Hair color is the primary state indicator: pink (normal), red (angry/possessed).
- Background contrast deliberately kept low so character pops.

**Idle vs action state differentiation:**
- Idle: Gentle breathing bob (2 frames, 1px up/down shift).
- Running: 6-frame cycle with exaggerated body lean.
- Dashing: Motion blur "squash-and-stretch" added frame.
- Death: Full white flash, then particle explosion — no intermediate "dying" animation.

**What "blocked/thinking/working" looks like:**
- Celeste does not have idle "thinking" states but the "hair" mechanic is instructive: the state of a single attribute (hair length/color) communicates the character's current capability.
- Blocked equivalent: Madeline standing on a platform with no obvious path = frustrated look (2-frame expression cycle).

**Outline style:** 1px hard outline, always black or very dark version of background color. Outline pixels are single-color and consistent around the entire sprite.

**Key takeaways for Agent Mission Control:**
- A single attribute change (e.g., glow color, accessory, or equipment) can signal state without redrawing the full sprite.
- For our 8 states: consider one "indicator element" (a screen glow, a floating symbol, tool in hand) that changes per state, while the base body stays similar.
- Color is the fastest state signal at small sizes. Assign each state a primary hue.
- 32px gives more room than 8px — use it for readable silhouette differences.

**URLs:** https://www.celestegame.com/ | Sprite analysis: https://github.com/NoelFB/Celeste

### 3. Dead Cells (Motion Twin / Evil Empire, 2018)

**Why it's relevant:** Dead Cells characters are 16–32px sprites with heavy animation (20–30 frames per action) and strong visual differentiation between states. The combat system requires instant readability of enemy state (idle / attacking / stunned / dead).

**Character size:** 16×16px base sprite, rendered at 2x–3x scale.

**Color palette approach:**
- Dark background (dungeon/sci-fi hybrid): near-black walls with high-contrast character pixels.
- Character palette uses 6–8 colors including a distinct "glow" or "aura" color that changes with state (blue for normal, orange for elite variants, red for cursed).
- Palette shifts are deliberate state signals, not decorative.

**Idle vs action state differentiation:**
- Idle: Heavy breathing animation with chest/shoulder motion. More frames (4–6) than FTL.
- Attacking: Exaggerated wind-up + fast strike. Limbs go "outside" the bounding box.
- Stunned: Whole sprite shakes/wobbles. Stars float above head.
- Dead: Immediate ragdoll + particle burst.

**What "blocked/thinking/working" looks like:**
- Stunned state is the closest "blocked" equivalent: character wobbles in place with visual indicator (floating stars/circles above head).
- No "thinking" state. Idle = potential state.
- Working (player attacking): All limbs engaged, clear motion direction.

**Outline style:** Soft outline on some sprites (1px, varies with character). Some characters use selective outline (only bottom/right edges) for a 3D light-source effect.

**Key takeaways for Agent Mission Control:**
- Floating icons above the character head work at 32px to signal state. Stars for confused, question mark for blocked, check for complete.
- "Aura" or glow effect around the sprite base can indicate urgency without changing silhouette.
- Differentiate active-working states (coding/testing/reading) through limb/body direction — typing faces screen, reading leans forward, testing looks at output.

**URLs:** https://store.steampowered.com/app/588650/Dead_Cells/

### 4. Hacknet (Team Fractal Alligator, 2015) + Uplink (Introversion, 2001)

**Why it's relevant:** Hacknet and Uplink establish the visual vocabulary for "hacker aesthetics." While neither uses pixel art sprites, their UI design principles are essential: terminal green on black, mission control aesthetics, monospace everything, visual urgency through text and sound.

**Character size:** N/A (no character sprites — pure UI games).

**Color palette approach:**
- Hacknet: `#00ff00` terminal green on `#000000` black. Secondary: `#ff4444` for errors, `#0099ff` for active connections.
- Uplink: Dark blue-gray with bright white terminal text. Red for detected/alerted state.
- Both: Monochrome primary palette with a single high-saturation accent for the "alert" state.

**Idle vs action state differentiation:**
- Neither has character sprites. State communicated through: text scroll speed, cursor blink rate, audio pitch.
- "Working": Text streams rapidly. Active animation.
- "Idle": Cursor blinks. Everything static.
- "Blocked/Detected": Screen flashes red. Audio alarm. Overlay text.
- "Completed": Success message appears. Transition animation.

**Key takeaways for Agent Mission Control:**
- Terminal green (`#00ff00`) and phosphor glow aesthetics are immediately readable as "hacker/tech."
- Our UI background should evoke terminal/mission control, even if sprites are colorful on top.
- "Working" states should feel kinetic — something moving (text scrolling, cursor blinking, body animating).
- "Blocked" must be immediately alarming — the visual should feel disruptive, not just different.

**URLs:** https://hacknet-os.com/ | https://www.introversion.co.uk/uplink/

### 5. Starbound (Chucklefish, 2016)

**Why it's relevant:** Starbound characters are 2–3 tiles tall in a space exploration context. Multiple animation states exist per race: idle, walking, running, jump, fall, hurt, dead. Most relevant: characters visually transform to show equipped tools (pickaxe, gun, etc.), which maps directly to our agent tool states.

**Character size:** ~20×36px, displayed at 2x scale (~40×72px).

**Color palette approach:**
- Per-species palette: humans use earth tones, Apex use dark browns/grays, Novakid use glowing neon.
- Equipment changes character silhouette visually — a character holding a mining laser looks different from one holding a gun.
- Background uses deep space black/navy with particle star fields.

**Idle vs action state differentiation:**
- Idle: Breathing bob (2 frames). Outfit animates slightly.
- Walking: 6-frame cycle. Tool swings at side.
- Interacting: Arm reaches out toward object. 2-frame loop.
- Hurt: Knockback frame. Character flashes red.

**What "blocked/thinking/working" looks like:**
- Interacting with an object maps well to our "coding" state: arms reach toward a screen/keyboard.
- Idle at a workbench = closest to "planning."
- No dedicated "blocked" state — hurt/knockback is the closest, but wrong tone.

**Outline style:** Thin 1px hard outline. Character outline is dark-colored (not always black — matched to character body dark tone).

**Key takeaways for Agent Mission Control:**
- Equipping a tool changes perceived purpose. For our 32px agent: holding a keyboard = coding, holding a magnifier = reading, holding a wrench = testing.
- Small handheld props at 32px can read clearly if contrasted against body color.
- Deep space backgrounds (near-black + subtle stars/gradients) are standard sci-fi trope for good reason — they create natural contrast for any sprite.

**URLs:** https://playstarbound.com/

### 6. Among Us (Innersloth, 2018)

**Why it's relevant:** Among Us characters are extremely simple (the crewmate silhouette is iconic at even 16px), yet they communicate state through minimal color and shape changes: walking, venting (impostor), emergency meeting, dead. The simplicity is instructive — you do not need complex anatomy.

**Character size:** ~12×18px at 1x, displayed at 2–4x scale.

**Color palette approach:**
- Single primary body color per player + slightly darker shadow + white visor.
- Body color is identity, not state. State communicated through: body position (upright vs slumped), overlay (ghost = dead), context (walking toward task vs walking toward body).
- Background is space station: dark floor tiles, subtle grid.

**Idle vs action state differentiation:**
- Walking: Legs alternate (2-frame cycle). Hat stays stationary.
- Task performing: Arms extend to side, body angles toward task.
- Killed: Body "split" into two halves, falling.
- Ghost: Full sprite slightly transparent/greyed.
- Emergency: All crewmates rush toward center — positional, not sprite-level.

**Key takeaways for Agent Mission Control:**
- Simpler anatomy = easier to read states. A "crewmate-style" astronaut body would work for our agent.
- Transparency/desaturation for "inactive" or "completed" states is a clean signal.
- Visor color change is one of the fastest readable state signals — consider a HUD visor that changes color per agent state.
- Among Us proves you can ship a beloved pixel art character at very low pixel counts if the silhouette is strong.

**URLs:** https://www.innersloth.com/games/among-us/

### 7. Stardew Valley (ConcernedApe, 2016)

**Why it's relevant:** Stardew characters are 16×32px and communicate detailed emotional states (happy, tired, upset, sick, festival-mode) through very subtle pixel changes: posture, eye expression, blush/sweat overlays. It is the closest reference for "character in a task context" expressing mental/emotional states rather than combat states.

**Character size:** 16×32px at 1x, displayed at 3x (48×96px effective).

**Color palette approach:**
- Warm, desaturated earth tones for characters. High-saturation accents for items/tools.
- Skin tone varies per character but stays within a coherent warm-analog palette.
- Status overlays use universal shorthand: Zzz for sleeping, hearts for happy, sweat drops for tired.

**Idle vs action state differentiation:**
- Idle: 2-frame breathing, feet static.
- Walking: 4-direction animation, arms swing.
- Chopping: Tool raised, arm-strike frame, sawdust particles.
- Tired (low energy): Head droops. Eyes half-closed (modified eye pixels). Slower movement.
- Happy (high mood): Smile expression (1px change to mouth), bouncy walk.

**What "blocked/thinking/working" looks like:**
- Thinking equivalent: NPC pause animation — they stop walking, face the player, expression changes.
- Tired/exhausted = closest "waiting/blocked" for our purposes.
- Working = tool animation, context-specific (watering, harvesting, etc.).

**Outline style:** Dark outlines with occasional absence on skin/face area to allow softer look. Consistent 1px on limbs and objects.

**Key takeaways for Agent Mission Control:**
- Emotional overlays (small floating symbols: Zzz, !, ?, heart) are universally understood visual shorthand.
- Eye expression changes radically affect perceived emotion even at 1px scale. Our 32px sprites should invest in expressive eyes.
- Posture encodes energy level: upright = active/working, slumped = waiting/failed, crossed-arms = blocked.

**URLs:** https://www.stardewvalley.net/

## PixelLab Style Notes

PixelLab AI (https://api.pixellab.ai/mcp/docs) is the generation tool we will use. Understanding its output characteristics before prompting is critical to producing usable assets.

### Output Quality by Size

**16x16px:**
- Detail is extremely limited. PixelLab can produce recognizable shapes but anatomy is impressionistic.
- Best results: abstract/geometric characters (robots, blobs, simple helmets). Humanoid anatomy is difficult.
- Recommended only for icons and status indicators, not agent characters.
- Shading: flat color + 1 shadow tone is the realistic ceiling. Gradient shading degrades rapidly.

**32x32px (our target):**
- PixelLab's strong zone. Enough pixels for a distinct head/body/legs silhouette + basic hand/tool.
- Facial expression is achievable: eyes (2px dots), mouth (1–2px line), eyebrow (1px).
- Recognizable props fit in this size: keyboard outline, magnifying glass, wrench, flag.
- Shading: 2–3 color tones per body region is achievable and looks good.
- Animation: PixelLab can generate keyframes but multi-frame generation needs careful prompting to maintain consistency.

**64x64px:**
- More detail, but output can look "muddy" if prompts are not precise. PixelLab tends to over-detail at this size.
- Good for one-off illustrations, not recommended for sprite sheets (file size and visual consistency issues).
- Our agent sprite should stay at 32px.

### What Prompt Descriptions Produce the Best Results

**Length:** Medium prompts (30–60 words) outperform both very short and very long prompts. Too short = generic output. Too long = conflicting style signals.

**Style-first:** PixelLab responds well to leading with style keywords before describing content. Example: "pixel art, 32x32, hard outline, limited palette, space theme — astronaut character..." outperforms "an astronaut character in pixel art style with hard outline..."

**Explicit negatives:** Always include "no anti-aliasing, no gradient shading, no blur" to prevent PixelLab from defaulting to hybrid raster-pixel style.

**Reference anchoring:** Mentioning known pixel art games as style anchors ("FTL-style crew member" or "Stardew Valley character proportions") can help anchor the aesthetic.

**Consistency for sprite sheets:** To generate multiple states that look like the same character, use an "anchor prompt" for the base character and then describe each state as a delta: "Same character as above, now [state-specific action]."

### Style Parameters for Space Aesthetics

**Outline style:** "hard 1px black outline" — produces the clearest readable sprites. Avoid "soft outline" or "no outline."

**Shading:** "flat shading with 2-tone shadow" — clean and consistent across frames. Avoid "dithered shading" for new prompts (dithering requires very precise prompting in PixelLab to avoid noise).

**Detail level:** "minimal detail, readable at small size" — tells PixelLab to prioritize silhouette clarity over texture.

**Background for character prompts:** Always prompt "transparent background" or "solid single-color background" — PixelLab's default background can interfere with sprite isolation.

**Animation frames:** Prompt each frame state separately with explicit description of pose, then composite in your sprite sheet. Do not prompt "animated sprite sheet" as PixelLab output spacing is inconsistent.

### Known Limitations

- **Color count:** PixelLab does not always honor a specific palette. You may need to post-process outputs with indexed palette reduction (e.g., limit to 8–16 colors via Aseprite or similar).
- **Consistency across generations:** Different runs will produce slightly different characters even with identical prompts. Use one "anchor" generation as the reference and manually fix subsequent states to match.
- **Faces at 32px:** PixelLab can generate face detail but it varies. Expect to manually touch up eyes and expression pixels.
- **Tool/prop detail:** Props at 32px often lose clarity. May need manual pixel editing to make a "keyboard" or "magnifier" clearly readable.
- **Animation timing:** PixelLab cannot produce proper animation timing data. Frame composition and timing values must be determined separately.

### Recommended PixelLab Parameters

```
width: 32
height: 32
outline: true
outline_width: 1
shading: flat
background: transparent
pixel_perfect: true
```

### Outline and Shading Recommendations

**Sprite outlines:** 1px hard black outline (`#000000` or `#0a0012` for palette harmony). No anti-aliasing.

**Shading:** 2-tone flat shading. Base color + 1 darker shadow tone (approximately 40% darker). No gradient, no dithering on first pass.

**Glow effect (optional):** For the current state, a 1px "halo" in the state color around the sprite outline is achievable in CSS (`filter: drop-shadow(0 0 3px ${stateColor)`)`) without modifying the sprite itself.

**Background on office canvas:** Deep dark blue-purple gradient (`#0a0a14` → `#0d0125`) with subtle dot-grid overlay at 10% opacity. Optional: very faint star field (white pixels at 5% density, randomized positions).

## State Communication at Small Scale

### The Challenge

At 32×32 pixels, the full canvas is 1024 pixels. A human face occupies roughly 8×8 pixels of that. The sprite must communicate one of 8 distinct states to a user who may be glancing at the Office Mode canvas with 10+ agents visible simultaneously.

Strategies in priority order:

### 1. Color (Fastest, Most Reliable)

The sprite border/glow color is readable at a glance even without looking at the sprite itself. Use a 1-2px colored border ring around each sprite, colored per state. This is the primary signal.

| State | Border Color | Animation |
|---|---|---|
| `planning` | `#a371f7` violet | Slow pulse (2s breathe) |
| `coding` | `#39d353` green | Fast pulse (0.5s) |
| `reading` | `#58a6ff` blue | Steady (no pulse) |
| `testing` | `#00d4ff` cyan | Rapid strobe (0.3s) |
| `waiting` | `#ffa657` amber | Very slow pulse (4s) |
| `blocked` | `#ff4444` red | Fast irregular flash |
| `completed` | `#3fb950` seafoam | Solid glow, no animation |
| `failed` | `#cc2222` dark red | Dim, no animation |

### 2. Floating Indicator Icon (Fast, Context-Dependent)

A small 8×8px icon floating above the character head provides a second signal layer. These are tiny and must be extremely simple.

| State | Icon | Visual |
|---|---|---|
| `planning` | Thought bubble | 3 dots (small → large), outline circle |
| `coding` | Keyboard | 3 short horizontal lines (key rows) |
| `reading` | Magnifier | Circle + handle |
| `testing` | Play button | Right-pointing triangle |
| `waiting` | Clock | Circle with 2 hands |
| `blocked` | Exclamation | Single vertical pixel + dot |
| `completed` | Checkmark | 3-pixel L-shape + upward |
| `failed` | X mark | 2 diagonal pixels |

### 3. Body Pose/Silhouette (Slower, But Persistent)

The sprite's body position encodes the emotional "weight" of the state. At 32px, major pose differences are readable.

| State | Posture | Key Visual Element |
|---|---|---|
| `planning` | Upright, hand to chin | Looking upward, chin-touch gesture |
| `coding` | Leaning forward | Arms extended, fingers toward screen |
| `reading` | Leaning back slightly | Head tilted, one hand raised (holding magnifier) |
| `testing` | Standing straight, tense | Eyes wide open, both arms at sides, ready posture |
| `waiting` | Slouching | Head drooped, arms loose at sides |
| `blocked` | Arms raised | Both hands up, "what happened?" pose |
| `completed` | Arms up (celebration) | V-arms, head raised, victory |
| `failed` | Head down, slumped | Curved spine, head drooping forward |

### 4. Animation Loop Type (Background Differentiator)

Even if a user isn't actively looking at a specific sprite, the animation pattern peripherally signals state.

| State | Animation | Frequency |
|---|---|---|
| `planning` | Slow head-tilt loop | 1–2fps |
| `coding` | Arm-type loop | 4–6fps (fast) |
| `reading` | Eye-scan loop | 2fps |
| `testing` | Rapid idle bounce | 4fps |
| `waiting` | Breathing only | 1fps (very slow) |
| `blocked` | Shake/fidget loop | 3fps (anxious) |
| `completed` | Static pose | No animation (finished) |
| `failed` | Slump loop (2-frame) | 1fps (defeated) |

### Concrete Recommendations per State

**`planning`** — Violet glow border, thought-bubble icon, hand-to-chin pose, slow 2s animation loop. Overall feeling: "I'm thinking something through."

**`coding`** — Green glow border, keyboard icon, arms-forward leaning pose, fast typing-arm animation. Overall feeling: "I'm actively producing."

**`reading`** — Blue glow border, magnifier icon, magnifier-in-hand pose, slow eye-scan animation. Overall feeling: "I'm absorbing information."

**`testing`** — Cyan glow border, play-button icon, tense-ready pose, bouncy fast animation. Overall feeling: "I'm running something."

**`waiting`** — Amber glow border, clock icon, slouched posture, breathing-only slow animation. Overall feeling: "I'm idle, not much happening."

**`blocked`** — RED glow border, exclamation icon, arms-up pose, anxious fidget animation. This is the most important state visually — an agent that needs human input should be IMMEDIATELY visible on the canvas. Consider adding a subtle screen flash or blink in addition to the CSS glow. Overall feeling: "I NEED ATTENTION."

**`completed`** — Seafoam glow border, checkmark icon, arms-up victory pose, NO animation (stillness signals completion). Overall feeling: "Done. Mission accomplished."

**`failed`** — Dark red dim border, X icon, slumped head-down pose, slow defeated animation. Subdued rather than alarming — failure is diagnostic, not urgent. Overall feeling: "Something went wrong."

```

limited 8-color palette, transparent background, no anti-aliasing, no gradient, no blur,
space station astronaut, small humanoid figure, full body visible,
wearing a dark spacesuit with colored visor, boots, simple proportions,
facing slightly left (3/4 view), upright neutral pose, large expressive eyes visible through visor,
FTL crew member proportions, Stardew Valley character style,
sci-fi pixel art, clean pixel art, readable silhouette
```

### Style Keywords That Work for Space Aesthetics

The following keywords have been found to reliably anchor PixelLab output toward space/sci-fi pixel art when included in prompts:

**Strongly effective:**
- `pixel art` (always first)
- `hard 1px black outline`
- `flat shading`
- `limited palette`
- `transparent background`
- `no anti-aliasing`
- `space station`
- `astronaut`
- `sci-fi`
- `FTL crew member style`

**Moderately effective:**
- `Stardew Valley proportions`
- `retro game sprite`
- `16-bit character`
- `small humanoid`
- `expressive eyes`
- `visor` (for astronaut helmet with visible face)
- `spacesuit`

**To avoid:**
- `cartoon` — pulls toward smooth, non-pixel style
- `detailed` — produces over-complex output at 32px
- `realistic` — defeats the pixel art aesthetic
- `gradient` — PixelLab will ignore the no-gradient instruction if gradient appears elsewhere
- `smooth` — same as above
- `high resolution` — confuses the target size

### Sprite Sheet Assembly Notes

When assembling the final sprite sheet (`agent-sheet.png`):

1. Generate each state as a separate PNG with transparent background.
2. Arrange states in rows: `planning` (row 0), `coding` (row 1), `reading` (row 2), `testing` (row 3), `waiting` (row 4), `blocked` (row 5), `completed` (row 6), `failed` (row 7).
3. Each frame is 32×32px. Even single-frame states get a 32px-wide slot.
4. Total sheet height: 8 states × 32px = 256px minimum (more if multi-frame animations are used).
5. Animation frames go left-to-right per row.
6. CSS `background-position` selects row: `background-position-y: calc(${stateIndex} * 32px)`.

### Color Palette Post-Processing

After generating sprites:
1. Import into Aseprite or similar tool.
2. Reduce to indexed color with a maximum 8-color palette per state, 16-color globally.
3. Ensure the state-color border color appears cleanly in the sprite itself (visor color should match the CSS state color exactly).
4. Export as 32-bit PNG (RGBA) with transparency preserved.
