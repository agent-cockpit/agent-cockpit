## Agent States (what the sprite must animate)

The `AgentAnimState` type drives all visual animations in Office Mode. There are exactly **8 states**, derived from `deriveAgentState()` in `spriteStates.ts`.

| State | CSS Class | Visual Emotion / Action | Trigger Priority & Conditions |
|---|---|---|---|
| `planning` | `sprite-planning` | Thinking, designing, strategizing | Priority 4: `session_start` event, OR `memory_write` event, OR `subagent_spawn` event |
| `coding` | `sprite-coding` | Actively writing/editing files | Priority 4: `tool_call` with write/edit/create/apply tool name, OR `file_change` event |
| `reading` | `sprite-reading` | Scanning files, searching codebase | Priority 4: `tool_call` with read/view/grep/search tool name, OR `memory_read` event |
| `testing` | `sprite-testing` | Running tests or executing commands | Priority 4: `tool_call` with test/run/exec/bash tool name |
| `waiting` | `sprite-waiting` | Idle, between actions | Priority 3: no last event available; OR after `subagent_complete`; OR default fallback |
| `blocked` | `sprite-blocked` | Paused, needs human decision | Priority 2 (high): `pendingApprovals > 0` on session; OR `approval_request` event |
| `completed` | `sprite-completed` | Task finished successfully | Priority 1 (highest): `session.status === 'ended'` |
| `failed` | `sprite-failed` | Error, parse failure | Priority 4: `provider_parse_error` event |

**State derivation logic (priority order):**
1. If `session.status === 'ended'` â†’ `completed` (overrides everything)
2. If `session.pendingApprovals > 0` â†’ `blocked` (urgent visual signal)
3. If no last event â†’ `waiting`
4. Switch on `lastEvent.type`:
   - `session_start` â†’ `planning`
   - `tool_call` â†’ regex-matched on tool name: reading/coding/testing/coding
   - `file_change` â†’ `coding`
   - `memory_read` â†’ `reading`
   - `memory_write` â†’ `planning`
   - `approval_request` â†’ `blocked`
   - `provider_parse_error` â†’ `failed`
   - `subagent_spawn` â†’ `planning`
   - `subagent_complete` â†’ `waiting`
   - default â†’ `waiting`

**Tool name classification (for `tool_call` events):**
- Regex `/read|view|grep|search/` â†’ `reading`
- Regex `/write|edit|create|apply/` â†’ `coding`
- Regex `/test|run|exec|bash/` â†’ `testing`
- No match â†’ `coding` (default)

**Sprite rendering:**
- Sprite sheet: `/sprites/agent-sheet.png`
- Sprite size: 32Ă—32 pixels
- Rendering: `imageRendering: pixelated`
- Each CSS class selects a row/frame in the sprite sheet
- Agent is draggable (dnd-kit) on an absolute-positioned canvas

## Event Types (what happens in a session)

The full union type is `NormalizedEvent` â€” a discriminated union of 11 event types.

| Event Name | Triggered By | Visual Meaning | UI Panels That Show It |
|---|---|---|---|
| `session_start` | Hook from Claude; CodexAdapter on thread open | Session begins â€” agent appears in Office Mode | SessionListPanel, OfficePage, TimelinePanel, SessionDetailPanel header |
| `session_end` | Claude hook on exit; Codex JSONL session end; daemon startup orphan cleanup | Session over â€” agent animates to completed | SessionListPanel (status dot â†’ gray), TimelinePanel |
| `tool_call` | Claude `PreToolUse` hook; Codex JSONL `function_call` line | Agent is doing something with a tool | TimelinePanel (expandable detail: toolName + JSON input), OfficePage (sprite state), AgentHoverCard (lastToolUsed) |
| `file_change` | Claude `PostToolUse` hook for Write/Edit tools | A file was created, modified, or deleted | TimelinePanel, DiffPanel (file tree + diff view) |
| `approval_request` | Claude `PreToolUse` hook for high/critical risk tools; Codex approval event | Agent wants to do something risky â€” NEEDS HUMAN RESPONSE | ApprovalInbox (card with action type, risk, proposed action, affected paths, why risky), TimelinePanel, OfficePage (sprite â†’ blocked) |
| `approval_resolved` | User clicks Approve/Deny/Always Allow in UI; OR timeout auto-deny | Human decided, agent unblocked | TimelinePanel (shows decision), OfficePage (sprite unblocks), ApprovalInbox (card removed optimistically) |
| `subagent_spawn` | Claude `SubagentStart` hook | Agent spawned a child agent | TimelinePanel |
| `subagent_complete` | Claude `SubagentStop` hook | Child agent finished | TimelinePanel |
| `memory_read` | Claude `PreToolUse` hook for Read tool on CLAUDE.md / auto memory | Agent consulted its memory | TimelinePanel, OfficePage (sprite â†’ reading) |
| `memory_write` | Claude `PostToolUse` hook when agent writes to memory file | Agent updated its memory â€” may need human approval if `suggested: true` | MemoryPanel (Pending Suggestions section), TimelinePanel |
| `provider_parse_error` | hookParser.ts or codexParser.ts when raw payload fails schema validation | Something went wrong in event parsing | TimelinePanel, OfficePage (sprite â†’ failed) |

**Base fields on every event:**
- `schemaVersion: 1` (always)
- `sessionId: string` (UUID)
- `timestamp: string` (ISO 8601)
- `sequenceNumber: number` (assigned by SQLite on insert â€” not part of payload JSON, merged in queries)

## Approval System (what needs attention UI)

### Action Types (what the agent wants to do)

| actionType | Description | Risk Range |
|---|---|---|
| `shell_command` | Execute a shell/bash command | medium â†’ critical |
| `file_change` | Write or edit a file | low |
| `network_access` | Make HTTP/network request | medium â†’ high |
| `sandbox_escalation` | Escalate privileges or break sandbox | (classified by risk classifier) |
| `mcp_tool_call` | Call an MCP-registered tool | low â†’ medium |
| `user_input` | Agent wants to ask the user a question | low |

### Risk Levels (visual urgency)

| riskLevel | Color Scheme | What Triggers It |
|---|---|---|
| `low` | Green border/text (`text-green-700 bg-green-100 border-green-300`) | file edits, user input, informational built-in tools |
| `medium` | Yellow border/text (`text-yellow-700 bg-yellow-100 border-yellow-300`) | regular bash, web fetch, unknown/MCP tools |
| `high` | Orange border/text (`text-orange-700 bg-orange-100 border-orange-300`) | curl/wget/ssh/git push/npm publish |
| `critical` | Red border/text (`text-red-700 bg-red-100 border-red-300`) | `rm -rf`, sudo, chmod 777, pipe-to-shell patterns |

**Human approval gate:** Only `high` and `critical` risk levels actually block the agent and require a human decision. `low` and `medium` are logged but not blocked.

### Approval Card Fields (shown in ApprovalInbox)

- `actionType` â€” formatted as title case ("Shell Command", "File Change", etc.)
- `riskLevel` â€” badge with color
- `proposedAction` â€” monospace text of what the agent wants to do (command, path, etc.)
- `affectedPaths` â€” list of file paths (optional)
- `whyRisky` â€” human-readable explanation of risk (optional)

### Decision Options

| Decision | UI Button | Backend Action |
|---|---|---|
| `approve` | Green "Approve" button | Hook server allows action; resolves event to `approved` |
| `deny` | Red "Deny" button | Hook server denies; resolves event to `denied` |
| `always_allow` | Blue "Always Allow" button | Inserts always_allow_rules row; resolves event to `always_allow` |
| `timeout` | (automatic) | Auto-deny after timeout; resolves event to `timeout`; no button |

**Optimistic UX:** After a decision is clicked, the approval card is immediately hidden (dismissed via local state), regardless of server confirmation.

### Approval DB Schema

```
approvals table:
  approval_id      TEXT PRIMARY KEY (UUID)
  session_id       TEXT NOT NULL
  status           TEXT NOT NULL DEFAULT 'pending'  -- pending|approved|denied|always_allow|timeout
  action_type      TEXT NOT NULL
  risk_level       TEXT NOT NULL
  proposed_action  TEXT NOT NULL
  affected_paths   TEXT  (JSON array, optional)
  why_risky        TEXT  (optional)
  created_at       TEXT NOT NULL
  decided_at       TEXT  (nullable)
  decision_reason  TEXT  (nullable)
```

## WebSocket Protocol

The daemon speaks a custom WebSocket protocol. The UI connects to `ws://localhost:3001`.

**Client â†’ Server messages:**
```json
{ "type": "approval_decision", "approvalId": "<uuid>", "decision": "approve|deny|always_allow" }
```

**Server â†’ Client messages:**
- On connect: catch-up replay â€” all events with `sequenceNumber > lastSeenSequence` sent as individual JSON event objects
- Live: every new `NormalizedEvent` broadcast to all connected clients as JSON
- Connection status drives UI: `disconnected | connecting | connected`

**WebSocket status visual signals:**
- `disconnected` â†’ Reconnecting banner (amber) in ApprovalInbox
- `connecting` â†’ (same, amber banner)
- `connected` â†’ Approval buttons enabled, normal operation
