# Phase 16.10: NPC Spawn Inside Map Fix — Research

**Researched:** 2026-04-13
**Domain:** Game state seeding, terrain walkability, coordinate system
**Confidence:** HIGH

## Summary

NPC characters currently spawn at world coordinates `(0, 0)` because the seeding formula uses a naive grid: `x: (i % COLS) * CELL, y: Math.floor(i/COLS) * CELL`. World coordinate `(0, 0)` maps to map tile `(-46, -43)` — the exact top-left corner of the bounding box, which is deep-space void (solid terrain ID 2). Every NPC 0 through 4 (first row) spawns in void, with later NPCs migrating progressively further into void still.

The fix is purely in the NPC seeding `useEffect` inside `OfficePage.tsx` (lines 214–230). No new infrastructure is needed: the tile walkability data is already loaded and `CollisionMap.isSolid()` can be added trivially. The recommended approach is to hardcode 12 verified spawn-slot pixel coordinates near the player start position — this is simpler, fully deterministic, and requires no async lookup at seed time. The player start at `(2080, 1920)` (tile `19,17`) is already on walkable floor (terrain ID 4) and does not need to change.

**Primary recommendation:** Replace the `(i % COLS) * CELL` formula with a fixed `SPAWN_SLOTS` array of 12 verified pixel-space positions, cycling with modulo when session count exceeds slot count, adding a small per-slot jitter offset to prevent perfect stacking.

## Coordinate System (Critical Context)

The map uses a tile-offset coordinate system:

| Concept | Value |
|---------|-------|
| `TILE_ORIGIN_X` (minX) | -46 |
| `TILE_ORIGIN_Y` (minY) | -43 |
| `TILE_SIZE` | 32 px |
| World pixel formula | `px = (tileX - (-46)) * 32 = (tileX + 46) * 32` |
| World `(0, 0)` = tile | `(-46, -43)` = deep-space void (solid ID 2) |
| Map playable area begins | approximately tile `(9, 9)` world |
| Player start | tile `(19, 17)` = px `(2080, 1920)` — terrain ID 4, walkable |

## CollisionMap API

`CollisionMap` (`packages/ui/src/game/CollisionMap.ts`) exposes:

```typescript
overlaps(x: number, y: number, w: number, h: number): boolean
```

This is the only public walkability query. It takes a world-pixel AABB and returns `true` if any corner hits a solid tile or solid object. To check if a single tile is walkable, call:
```typescript
collisionMap.overlaps(px, py, 1, 1)  // returns true = solid/blocked
```

There is no `isTileWalkable(tx, ty)` helper today — but one can be derived from `overlaps`. The `solidTiles` private field is a `Set<string>` keyed as `"${tx},${ty}"` where `tx = Math.floor(px / 32)`.

**Important:** `CollisionMap` is populated asynchronously in `OfficePage.tsx` — terrain and objects load after a `fetch()`. The NPC seeding `useEffect` runs synchronously on `sessions` changes. If the fix relies on runtime `overlaps()` calls, there is a race condition: the collision map may not yet be loaded when the first sessions arrive. This is why hardcoded spawn slots are preferable over runtime scanning.

The ONLY file that needs modification is `OfficePage.tsx`. The seeding `useEffect` (lines 214–230):

```typescript
    }
  })
  // ...
}, [sessions])
```

```typescript
// FIXED (recommended):
const SPAWN_SLOTS: ReadonlyArray<{ x: number; y: number }> = [
  { x: 1952, y: 1792 }, { x: 2016, y: 1792 }, { x: 2112, y: 1792 }, { x: 2240, y: 1792 },
  { x: 2080, y: 1920 }, { x: 2176, y: 1920 }, { x: 2272, y: 1920 },
  { x: 1984, y: 2016 }, { x: 2112, y: 2016 }, { x: 2240, y: 2016 },
  { x: 1952, y: 2144 }, { x: 2080, y: 2144 },
] as const

const SPAWN_JITTER = 16  // px offset to prevent exact stacking when cycling

useEffect(() => {
  sessions.forEach((session, i) => {
    if (!gameState.npcs[session.sessionId]) {
      const slot = SPAWN_SLOTS[i % SPAWN_SLOTS.length]
      const cycle = Math.floor(i / SPAWN_SLOTS.length)
      gameState.npcs[session.sessionId] = {
        x: slot.x + (cycle > 0 ? (cycle % 3) * SPAWN_JITTER : 0),
        y: slot.y + (cycle > 0 ? Math.floor(cycle / 3) * SPAWN_JITTER : 0),
      }
    }
  })
  // clean up ended sessions...
}, [sessions])
```

## Common Pitfalls

### Pitfall 1: The Async CollisionMap Race
**What goes wrong:** Calling `collisionMap.overlaps()` inside the seeding `useEffect` before terrain data has loaded — `collisionMap` is empty at that point and everything appears walkable but for wrong reasons.
**Why it happens:** `fetch('/map/terrain-map.json')` is async; the first session may arrive before it resolves.
**How to avoid:** Use hardcoded spawn slots. If runtime checking is ever desired, gate it with a `collisionMapReady` ref flag set in the `.then()` handler.

### Pitfall 2: Overwriting Existing NPC Positions
**What goes wrong:** Forgetting the `if (!gameState.npcs[session.sessionId])` guard, causing all NPCs to teleport to spawn slots every time `sessions` changes.
**Why it happens:** The `useEffect` runs on every `sessions` update.
**How to avoid:** The existing guard `if (!gameState.npcs[session.sessionId])` must be preserved. Success criterion 5 requires existing positions to be untouched.

### Pitfall 3: Session Index Drift
**What goes wrong:** Using `sessions.indexOf(session)` or `sessions.forEach((session, i) => ...)` — the index `i` changes as sessions are added/removed, potentially assigning a different slot to an existing NPC if the guard is incorrectly removed.
**How to avoid:** The guard `if (!gameState.npcs[session.sessionId])` prevents re-assignment regardless of index drift.

### Pitfall 4: SPAWN_SLOTS Placement Affecting Tests
**What goes wrong:** Placing `SPAWN_SLOTS` inside the component function — it gets re-created on every render, which is fine functionally but wasteful and makes snapshot tests harder to freeze.
**How to avoid:** Module-level `const` declaration.

## Architecture Patterns

### Existing Seeding Pattern (Preserved Structure)

The seeding `useEffect` already handles cleanup (removing ended sessions). The fix must preserve:
1. `if (!gameState.npcs[session.sessionId])` guard — existing positions not overwritten
2. Cleanup loop: `activeIds` set + `delete gameState.npcs[id]` for ended sessions

### Jitter Strategy

When `sessions.length > SPAWN_SLOTS.length` (12), agents cycle through slots. To prevent exact overlap:
- `cycle = Math.floor(i / SPAWN_SLOTS.length)` — which full cycle through slots this is
- Small pixel offset per cycle avoids perfect stacking while keeping agents on floor tiles
- Offset of 16px is safe: slots are 32-64px apart, so a 16px jitter stays within the walkable tile

## Code Examples

### Tile-to-Pixel Conversion (Reference)
```typescript
// From CollisionMap.ts constants:
const TILE_ORIGIN_X = -46  // map.json boundingBox.minX
const TILE_ORIGIN_Y = -43  // map.json boundingBox.minY

// Pixel for tile (tx, ty):
```
