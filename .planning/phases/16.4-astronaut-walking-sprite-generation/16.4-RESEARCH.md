# Phase 16.4: Astronaut Walking Sprite Generation — Research

**Researched:** 2026-04-12
**Domain:** PixelLab MCP API, sprite sheet assembly, game sprite integration
**Confidence:** HIGH

---

## Summary

The current astronaut sprite sheet has no walking frames. The player walk animation implemented in Phase 16.2 repurposes the `blocked` state rows (rows 8-15) — which contain an 8-frame "fight-stance-idle" animation, not a walk cycle. The visual result is the player character doing a fighting stance shuffle instead of walking. Phase 16.4 must generate proper walk cycle frames using PixelLab's `animate_character` endpoint, assemble them into the astronaut sprite sheet, and update the render code to point at the correct row group.

The Phase 13 pipeline (`scripts/build-spritesheet.ts`, `scripts/validate-sprites.ts`, raw frame storage at `assets/raw/astronaut/`) is fully functional and reusable. The existing astronaut `character_id` (`c76f4e82-7eec-4443-96fa-501def9d1656`) can be used directly — no need to regenerate the base character. Budget is comfortable: 1438 images remain unspent from the 2000-image total.

The key design decision is whether to add a new walk row group to the sprite sheet (expanding from 32 to 40 rows) or replace the `blocked` rows with walk frames. Adding new rows is preferable because it preserves the existing blocked animation and requires only a row offset update in the render code. The sprite sheet would grow from 576×2048 to 576×2560 (40 rows × 64px).

**Primary recommendation:** Generate walk animation for the astronaut using `walking-8-frames` template (8 directions, 8 frames), add as new row group at offset 32 in the sprite sheet, and update `OfficePage.tsx` to use `STATE_ROW_OFFSET.walk` (= 32) instead of `STATE_ROW_OFFSET.blocked` for player walking. The astronaut is the only character used as the player sprite — no other characters need walk frames in this phase.

## Sprite Sheet Architecture Decision

### Option A: Add walk rows at offset 32 (RECOMMENDED)

Expand sprite sheet from 32 rows to 40 rows. Walk rows occupy rows 32-39 (8 directions × 1 animation state).

**Sprite sheet layout after Phase 16.4:**
```
Rows 0-7:   idle (4 frames)
Rows 8-15:  blocked (8 frames) — preserved, NPCs still use this
Rows 16-23: completed (9 frames)
Rows 24-31: failed (7 frames)
Rows 32-39: walk (8 frames)  ← NEW
```

**Sheet dimensions:** 576×2560px (9 cols × 64 = 576; 40 rows × 64 = 2560)

**Code change required in `OfficePage.tsx`:**
```ts
// Before:
const stateOffset = gameState.player.animTime > 0 ? STATE_ROW_OFFSET.blocked : STATE_ROW_OFFSET.idle

// After:
const stateOffset = gameState.player.animTime > 0 ? STATE_ROW_OFFSET.walk : STATE_ROW_OFFSET.idle
```

**Where to define `STATE_ROW_OFFSET.walk`:** Add to `spriteStates.ts`:
```ts
export const STATE_ROW_OFFSET: Record<AnimationState, number> = {
  idle: 0,
  blocked: 8,
  completed: 16,
  failed: 24,
  walk: 32,   // NEW
}
```

**Why recommended:** Preserves the blocked animation (NPCs use it; it still works correctly). Minimal code change. Consistent with existing pattern.

### Option B: Replace blocked rows with walk rows

Overwrite rows 8-15 with walk frames. NPCs assigned to `blocked` state would then show walking instead of the fight stance.

**Not recommended.** The blocked state is used in `drawAgentSprite()` for NPCs when `pendingApprovals > 0`. Replacing it breaks NPC visual differentiation.

## Architecture Patterns

### Pattern 1: Reuse Phase 13 Pipeline Verbatim

The `build-spritesheet.ts` script is designed for extension. Adding `'walk'` to `ANIM_STATES` is the only change needed — no structural modifications.

```bash
# After walk frames are downloaded to assets/raw/astronaut/walk/
npx tsx scripts/build-spritesheet.ts astronaut
# Produces: 576x2560px sheet with 40 rows
```

### Pattern 2: AnimationState Type Extension

`AnimationState` in `spriteStates.ts` must include `'walk'`. This is used by `STATE_ROW_OFFSET` and potentially by `DrawAgentSpriteOptions`.

```ts
// spriteStates.ts — add 'walk' to the union:
export type AnimationState = 'idle' | 'blocked' | 'completed' | 'failed' | 'walk'

export const STATE_ROW_OFFSET: Record<AnimationState, number> = {
  idle: 0,
  blocked: 8,
  completed: 16,
  failed: 24,
  walk: 32,
}
```

Note: `COLOR_STATE_TO_ANIMATION` maps player states to `AnimationState`. Walking is not an `AgentAnimState` (NPCs don't walk in Phase 16.4) — the `walk` row is accessed directly in `OfficePage.tsx` player render only.

### Pattern 3: Visual Checkpoint Before Integration

After generating walk frames, open `astronaut-sheet.png` in an image viewer to confirm rows 32-39 show a believable gait cycle before updating the render code. The checkpoint is blocking — if the template produces a wrong animation (e.g., `walking-8-frames` returns a run rather than walk), retry with `walk` or `walking` templates before proceeding.

### Anti-Patterns to Avoid

- **Queuing all 8 directions in one call and immediately downloading:** PixelLab has an 8-slot limit. Wait for the animation job to complete before downloading frames.
- **Modifying `blocked` rows:** NPCs depend on the blocked animation for visual state differentiation.
- **Extending all 10 characters:** Only the astronaut is used as the player sprite. Other characters are NPCs that don't need walk rows.
- **Hardcoding walk row as 32 in OfficePage.tsx:** Always reference `STATE_ROW_OFFSET.walk` from `spriteStates.ts` — keeps the layout definition in one place.

## Common Pitfalls

### Pitfall 1: `walking-8-frames` template may not exist
**What goes wrong:** PixelLab returns an error for `template_animation_id: "walking-8-frames"`.
**Why it happens:** Template names are only partially documented; some names in the WebSearch list may be incorrect.
**How to avoid:** Use fallback chain: `walking-8-frames` → `walk` → `walking` → custom `action_description`. For custom mode: `action_description: "natural walking cycle, feet alternating, arms swinging"`.
**Warning signs:** MCP call returns status 422 or error mentioning invalid template_id.

### Pitfall 2: Walk frame count differs from WALK_FRAME_COUNT = 8
**What goes wrong:** Template produces 4 or 6 frames, but `WALK_FRAME_COUNT = 8` in `PlayerInput.ts` causes the modulo to cycle over empty frames.
**Why it happens:** Different templates have different frame counts. `walking-4-frames` would produce 4, `walking-6-frames` would produce 6.
**How to avoid:** After download, count actual frames in `assets/raw/astronaut/walk/south/`. If count ≠ 8, update `WALK_FRAME_COUNT` constant to match, and update `astronaut-manifest.json` `walk` entry accordingly.
**Warning signs:** Player animation has a visible "blank" or "freeze" frame at end of cycle.

### Pitfall 3: build-spritesheet.ts ANIM_STATES order matters
**What goes wrong:** If `'walk'` is inserted in the middle of `ANIM_STATES`, existing rows shift. All NPCs using `blocked`/`completed`/`failed` would render from the wrong row.
**Why it happens:** The script assembles rows in `ANIM_STATES` order — row group for `ANIM_STATES[0]` occupies rows 0-7, `ANIM_STATES[1]` rows 8-15, etc.
**How to avoid:** Always append `'walk'` at the END of `ANIM_STATES`. Verify the final sheet with `validate-sprites.ts` and visual inspection.
**Warning signs:** NPCs show walking animation instead of blocked/completed/failed states.

### Pitfall 4: 8-slot limit on PixelLab concurrent jobs
**What goes wrong:** If any other animations are still processing when `animate_character` is called, it returns HTTP 423.
**Why it happens:** PixelLab limits concurrent animation jobs per character.
**How to avoid:** Check `get_character` shows no pending jobs before calling `animate_character`. Wait for prior animations to complete.

### Pitfall 5: stateOffset still uses `blocked` after integration
**What goes wrong:** Player walks but shows fight stance because `OfficePage.tsx` still references `STATE_ROW_OFFSET.blocked`.
**Why it happens:** The PLAN.md for 16.4-02 must explicitly call out this code change — it's easy to miss as a one-liner.
**How to avoid:** The integration plan must include a concrete line-by-line diff for `OfficePage.tsx` line 115.
