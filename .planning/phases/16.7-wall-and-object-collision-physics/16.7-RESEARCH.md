# Phase 16.7: Wall and Object Collision Physics - Research

**Researched:** 2026-04-12
**Domain:** 2D tile-based collision detection, Canvas game loop, TypeScript
**Confidence:** HIGH

## Summary

The project uses a custom Canvas 2D game engine (no third-party physics library). Movement is handled by `movePlayer()` in `PlayerInput.ts`, which currently applies raw delta to player position clamped only at world edges. The map is a 3232×3232px pre-rendered composite (`map-composite.png`) derived from `terrain-map.json` (tile-space coords) and `objects/manifest.json` (pixel-space bounding boxes).

Collision physics must be implemented at the movement layer: before committing the new player position, test the candidate position against a collision map. The collision map is built once at load time from two sources: (1) solid terrain tiles (terrainId 3 = walls, terrainId 2 = deep space void exterior) read from `terrain-map.json`, and (2) solid map objects (furniture/obstacles) read from `objects/manifest.json`. All checks are pure pixel-space AABB (axis-aligned bounding box) tests — no physics library is warranted.

**Primary recommendation:** Build a `CollisionMap` module that pre-computes a `Set<string>` of occupied pixel-tile keys from terrain + object data at startup, then modify `movePlayer()` to accept a `CollisionMap` and reject positions whose player AABB overlaps any blocked cell. Slide-along-wall behavior (test X and Y axes independently) gives smooth wall-glide typical of top-down RPGs.

### Pattern 1: Tile-Space Collision Grid (Sparse Set)

**What:** Convert solid terrain cells (tile-space) to pixel-space blocked tiles. Store as `Set<string>` keyed by `"${tileX},${tileY}"` where each tile is 32×32px. Query by dividing the player AABB corners into tile coords.

**When to use:** Map is large (101×101 = 10201 tiles), but only ~214 wall cells and ~4214 void cells are solid. A sparse Set is O(1) lookup, avoids a 10201-entry array, and is trivially serializable.

**Coordinate transform:**
- `terrain-map.json` stores tile-space x/y with origin offset `minX=-46, minY=-43`
- Pixel = `(tileX - (-46)) * 32` = `(tileX + 46) * 32`
- This is exactly the same `tileToPixel()` transform already in `TilemapRenderer.ts`

```typescript
// Source: existing TilemapRenderer.ts tileToPixel()
const TILE_SIZE = 32
const TILE_ORIGIN_X = -46
const TILE_ORIGIN_Y = -43

function tileToPixelCoord(tx: number, ty: number) {
  return {
    px: (tx - TILE_ORIGIN_X) * TILE_SIZE,
    py: (ty - TILE_ORIGIN_Y) * TILE_SIZE,
  }
}
```

**Solid terrain IDs:**
- `terrainId: 3` — "solid interior wall" (214 cells) — BLOCKED
- `terrainId: 2` — "deep space void exterior" (4214 cells) — BLOCKED
- `terrainId: 1` — hull floor — walkable
- `terrainId: 4` — patrol line — walkable
- `terrainId: 5` — scuffed panels — walkable
- `defaultTerrain: 0` — unspecified tiles (vast majority of the map) — walkable (open floor)

### Pattern 2: Object AABB Registry

**What:** Map objects have pixel-space `boundingBox {x, y, width, height}` in `objects/manifest.json`. Store solid-object AABBs as an array of `Rect` structs. On movement test, check player AABB against each object rect.

**Which objects are solid:**
- Characters (NPCs in manifest) — these are decorative art assets, NOT game-engine NPCs. They appear to be layer 2–11 items. They are visually in the map composite but do NOT represent interactive entities. Decision required: treat all non-character objects as solid (desks, pool table, piano, arcade cabinets, slot machines, couch, bed, wardrobe, server rack, altar, idol, crystal ball, etc.)
- Character objects (layers 2–11, descriptions start with "Character:") are sprite decorations — they should NOT block movement (they are ambient, not interactive furniture)
- Furniture/objects (layers 13+, non-character descriptions) — SOLID

**Filter heuristic:** Objects whose `description` starts with `"Character:"` are non-solid ambient sprites. All others with `visible: true` are solid furniture/obstacles.

**Total solid objects:** ~20 furniture items (pool table, piano, arcade cabinets × 3, slot machines, couch, bed, wardrobe, server rack, alter, idol, crystal ball, desk × 3, personal desk)

### Pattern 3: Slide-Along-Wall (Axis-Separated Movement)

**What:** Test X and Y movement independently. If moving diagonally and blocked on X, still apply the Y component (and vice versa). This produces natural wall-sliding instead of a hard stop.

**When to use:** Always — it is the standard pattern for top-down RPGs and prevents the frustrating "stuck in a corner" experience.

```typescript
// Pseudo-code for slide behavior in movePlayer()
const newX = player.x + dx * dist
const newY = player.y + dy * dist

const xBlocked = collisionMap.overlaps(newX, player.y, PLAYER_W, PLAYER_H)
const yBlocked = collisionMap.overlaps(player.x, newY, PLAYER_W, PLAYER_H)

player.x = xBlocked ? player.x : clamp(newX, 0, WORLD_W - PLAYER_W)
player.y = yBlocked ? player.y : clamp(newY, 0, WORLD_H - PLAYER_H)
```

### Pattern 4: Player AABB Shrink

**What:** The player sprite is 64×64px but the visual character occupies roughly the lower 48×32px of the sprite (feet area). Using the full 64×64 bounding box creates an invisible wall gap. A shrunken hitbox (e.g., 32×32 centered at the feet) feels more natural.

**Recommended hitbox:** 32px wide × 32px tall, offset 16px from sprite top-left on X, 32px on Y.

```typescript
const PLAYER_HITBOX = { offsetX: 16, offsetY: 32, w: 32, h: 32 }
```

### Anti-Patterns to Avoid

- **Full 2D array for collision grid:** Wastes 10K+ entries for mostly-empty open floor. Use sparse Set.
- **Recalculating collision map every frame:** Build once at load, use for all frames.
- **Checking every object every pixel tick:** 20 objects is trivial but keep as simple array loop — no spatial acceleration needed at this scale.
- **Moving player to "last valid position" on collision:** Causes jitter. Use slide (axis-separated) instead.
- **Blocking on void/space tiles with tight margins:** Player start is at pixel (1472, 1376) = tile (46+(-46), 43+(-43)) = tile (0,0). Verify the starting area is walkable before final testing.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Swept AABB (continuous collision) | custom swept-AABB resolver | Discrete AABB per frame | Discrete is sufficient at 60 FPS with PLAYER_SPEED=120px/s (2px/frame max) — tunneling impossible |
| Spatial acceleration (quadtree) | QuadTree class | Flat array of ~20 objects | Only 20 collidable objects; O(n) loop is faster than tree overhead |
| Third-party physics (Matter.js, Rapier) | Full physics integration | Custom tile+AABB check | Adds WebAssembly/heavy dependency for a top-down RPG with no physics simulation needed |

## Common Pitfalls

### Pitfall 1: Coordinate System Mismatch
**What goes wrong:** terrain-map.json uses tile-space coords with origin `minX=-46, minY=-43`. Object manifest uses pixel-space coords measured from top-left of the map image (i.e., already in pixel-space). These are different reference frames.
**Why it happens:** Two different data sources with different origins.
**How to avoid:** Terrain tiles: apply `tileToPixel(tx, ty)` transform. Object bounding boxes: use `{x, y}` directly — they are already in pixel-space relative to the map's top-left (0,0).
**Warning signs:** Player hits invisible walls offset from visible walls, or passes through visible walls.

### Pitfall 2: Default Terrain Handling
**What goes wrong:** `terrain-map.json` is sparse — only 5107 cells are explicitly stored. The `defaultTerrain: 0` means unspecified tiles are terrain type 0 (not in the terrains list = open floor). Treating default as solid would block movement everywhere.
**Why it happens:** Assuming all unspecified terrain is a wall.
**How to avoid:** Only tiles explicitly listed with `terrainId: 2` or `terrainId: 3` are solid. Any tile NOT in the sparse set is walkable.

### Pitfall 3: Player Start Position in Void
**What goes wrong:** Player starts at pixel (1472, 1376) — verify this maps to a walkable floor tile. If it is in a void cell, the player spawns trapped.
**Why it happens:** Start position was set to `tile (46*32, 43*32)` based on a bounding box comment; actual terrain at that pixel needs verification.
**How to avoid:** Write a test that asserts the player start position is not in a blocked tile. If it is, adjust start position in `GameState.ts`.

### Pitfall 4: movePlayer Signature Change
**What goes wrong:** `movePlayer()` is a pure function called from OfficePage with `(player, keys, deltaMs)`. Adding a `CollisionMap` parameter changes the signature — the call site in `OfficePage.tsx` and all `PlayerInput.test.ts` tests must be updated.
**Why it happens:** Existing tests call `movePlayer()` without collision data.
**How to avoid:** Make `CollisionMap` an optional 4th parameter (or pass `null` to disable collision) for backward compatibility with tests that don't need a map.

### Pitfall 5: Object BoundingBox Coordinate Space
**What goes wrong:** Objects in `manifest.json` use pixel coords that appear to be in a different coordinate space than expected — they were placed in the map editor's pixel-space. The map editor's world appears to run from (0, 0) to (worldW, worldH), matching the rendered PNG. However, objects like the pool table sit at `{x:909, y:542}` which is well within the 3232px world — these are pixel-space world coordinates that can be used directly for AABB tests.
**Why it happens:** Two sources (terrain: tile-space with offset, objects: pixel-space from (0,0)) need different handling.
**How to avoid:** For objects, use `boundingBox.{x,y,width,height}` directly as pixel-space coordinates — no transform needed.

## Code Examples

### CollisionMap Module (recommended implementation)

```typescript
// packages/ui/src/game/CollisionMap.ts

export interface Rect { x: number; y: number; w: number; h: number }

const TILE_SIZE = 32
const TILE_ORIGIN_X = -46  // from map.json mapConfig.boundingBox.minX
const TILE_ORIGIN_Y = -43  // from map.json mapConfig.boundingBox.minY

// Terrain IDs that block movement
const SOLID_TERRAIN_IDS = new Set([2, 3])  // void exterior + interior wall

interface TerrainCell { x: number; y: number; terrainId: number }
interface TerrainMapData { defaultTerrain: number; cells: TerrainCell[] }
interface ObjectEntry { boundingBox: { x: number; y: number; width: number; height: number }; description: string; visible: boolean }

export class CollisionMap {
  private solidTiles = new Set<string>()
  private solidObjects: Rect[] = []

  loadTerrain(data: TerrainMapData): void {
    this.solidTiles.clear()
    for (const cell of data.cells) {
      if (SOLID_TERRAIN_IDS.has(cell.terrainId)) {
        const px = (cell.x - TILE_ORIGIN_X) * TILE_SIZE
        const py = (cell.y - TILE_ORIGIN_Y) * TILE_SIZE
        // Key by tile index (not pixel) — each tile covers 32x32
        const tx = Math.floor(px / TILE_SIZE)
        const ty = Math.floor(py / TILE_SIZE)
        this.solidTiles.add(`${tx},${ty}`)
      }
    }
  }

  loadObjects(objects: ObjectEntry[]): void {
    this.solidObjects = []
    for (const obj of objects) {
      if (!obj.visible) continue
      if (obj.description.startsWith('Character:')) continue  // ambient sprites, not obstacles
      const { x, y, width, height } = obj.boundingBox
      this.solidObjects.push({ x, y, w: width, h: height })
    }
  }

  // Returns true if the given AABB overlaps any solid tile or object
  overlaps(x: number, y: number, w: number, h: number): boolean {
    // Check tiles: test all 4 corners + center
    const corners = [
      { cx: x,       cy: y       },
      { cx: x + w-1, cy: y       },
      { cx: x,       cy: y + h-1 },
      { cx: x + w-1, cy: y + h-1 },
    ]
    for (const { cx, cy } of corners) {
      const tx = Math.floor(cx / TILE_SIZE)
      const ty = Math.floor(cy / TILE_SIZE)
      if (this.solidTiles.has(`${tx},${ty}`)) return true
    }
    // Check objects (AABB vs AABB)
    for (const obj of this.solidObjects) {
      if (x < obj.x + obj.w && x + w > obj.x &&
          y < obj.y + obj.h && y + h > obj.y) {
        return true
      }
    }
    return false
  }
}
```

### Modified movePlayer Signature

```typescript
// PlayerInput.ts — updated signature (CollisionMap is optional for test backward-compat)
import type { CollisionMap } from './CollisionMap.js'

export const PLAYER_HITBOX = { offsetX: 16, offsetY: 32, w: 32, h: 32 }

export function movePlayer(
  player: { x: number; y: number; direction: string; animTime: number },
  keys: ReadonlySet<string>,
  deltaMs: number,
  collisionMap?: CollisionMap,
): void {
  // ... existing dx/dy/dist calculation unchanged ...

  const newX = player.x + dx * dist
  const newY = player.y + dy * dist

  if (collisionMap) {
    const hx = newX + PLAYER_HITBOX.offsetX
    const hy = newY + PLAYER_HITBOX.offsetY
    const xBlocked = collisionMap.overlaps(hx, player.y + PLAYER_HITBOX.offsetY, PLAYER_HITBOX.w, PLAYER_HITBOX.h)
    const yBlocked = collisionMap.overlaps(player.x + PLAYER_HITBOX.offsetX, hy, PLAYER_HITBOX.w, PLAYER_HITBOX.h)
    player.x = xBlocked ? player.x : Math.max(0, Math.min(newX, WORLD_W - 64))
    player.y = yBlocked ? player.y : Math.max(0, Math.min(newY, WORLD_H - 64))
  } else {
    player.x = Math.max(0, Math.min(newX, WORLD_W - 64))
    player.y = Math.max(0, Math.min(newY, WORLD_H - 64))
  }

  // ... existing direction + animTime logic unchanged ...
}
```
