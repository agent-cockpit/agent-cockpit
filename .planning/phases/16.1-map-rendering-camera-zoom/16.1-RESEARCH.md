## Summary

Phase 16.1 replaces the CSS background floor tile and 1920×1440 world with a real 3232×3232px tilemap rendered on Canvas. The map export package contains all needed assets: 7 terrain types described via Wang tilesets (each a 4×4 PNG grid = 16 tiles at 32px), a sparse terrain-map (tile coords in range −46..+54 x, −43..+57 y), a transition-map (~725KB JSON, edge-level Wang tile selections per cell), one inpainting overlay PNG, and 30 placed objects with pixel-space bounding boxes.

The rendering architecture must address three constraints: (1) 10,201 tiles (101×101) cannot be redrawn per frame at 60fps without a pre-render; (2) the 2x zoom must apply uniformly to terrain, objects, NPCs, and player via `ctx.scale(2,2)` before drawing world-space content; (3) bounds clamping must operate in zoomed screen space (effectively clamp against 6464×6464 equivalent).

The correct approach is a single `OffscreenCanvas` of 3232×3232 built once at engine load (drawing all terrain tiles, overlay, and objects in layer order), then blitting only the viewport slice to the main canvas each frame inside a `ctx.scale(2,2)` transform.

**Primary recommendation:** Pre-render all static content (terrain + overlay + objects) to a 3232×3232 OffscreenCanvas at load. Each frame: `ctx.scale(2,2)` → `ctx.drawImage(offscreen, -cam.x, -cam.y)` → draw dynamic NPCs/player in world space → `ctx.restore()`.

### Pattern 1: Offscreen Canvas Pre-render
**What:** Build a 3232×3232 OffscreenCanvas once at engine start. Sequence: fill with base terrain → draw transition tiles → blit overlay → draw objects sorted by `layer` ascending.
**When to use:** Always — this is the correct approach for static tilemap content with 10k+ tiles.
**Example:**
```typescript
// TilemapRenderer.ts
export class TilemapRenderer {
  private offscreen: OffscreenCanvas
  private ready = false

  async load(): Promise<void> {
    const [mapData, terrainMap, transitionMap, objManifest] = await Promise.all([
      fetch('/map/map.json').then(r => r.json()),
      fetch('/map/terrain-map.json').then(r => r.json()),
      fetch('/map/transition-map.json').then(r => r.json()),
      fetch('/map/objects/manifest.json').then(r => r.json()),
    ])
    this.offscreen = new OffscreenCanvas(3232, 3232)
    const ctx = this.offscreen.getContext('2d')!
    // ... build terrain lookup, draw tiles, overlay, objects ...
    this.ready = true
  }

  blit(ctx: CanvasRenderingContext2D, camX: number, camY: number): void {
    if (!this.ready) return
    // Called inside ctx.save() / ctx.scale(2,2) block in render()
    ctx.drawImage(this.offscreen, -camX, -camY)
  }
}
```

### Pattern 2: 2x Zoom Render Transform
**What:** Wrap the entire world-space draw sequence in `ctx.save()` / `ctx.scale(zoom, zoom)` / `ctx.restore()`. All world coordinates (tiles, objects, NPCs, player) are passed in world-space; the scale handles projection to screen-space.
**When to use:** In OfficePage render() — wraps tilemap blit AND NPC/player draws.
**Example:**
```typescript
render() {
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  ctx.save()
  ctx.scale(zoom, zoom)  // zoom = 2
  tilemapRenderer.blit(ctx, gameState.camera.x, gameState.camera.y)
  // draw NPCs (existing code, unchanged coordinates)
  // draw player (existing code, unchanged coordinates)
  ctx.restore()
}
```

### Pattern 3: Zoom-adjusted Camera Bounds
**What:** Camera target and clamping must account for zoom. The visible world region in world-space at 2x zoom is `viewportW/zoom` × `viewportH/zoom`. The camera x/y (top-left of the visible region in world-space) must be clamped so the bottom-right of the viewport never exceeds world bounds.
**Example:**
```typescript
// In update(), set camera target to center player in world-space viewport
const visW = cam.viewportW / zoom   // e.g. 800/2 = 400
const visH = cam.viewportH / zoom
cam.targetX = player.x - visW / 2
cam.targetY = player.y - visH / 2

// updateCamera bounds: maxX and maxY stay as world-space 3232
// clamp formula already is: cam.x <= maxX - viewportW
// BUT viewportW in CameraState is the screen dimension, not world-space.
// SOLUTION: Pass effective bounds adjusted for zoom, OR store viewportW in world-space.
// Cleanest: compute zoomed viewport size in update() before calling updateCamera():
updateCamera(cam, {
  minX: 0, minY: 0,
  maxX: WORLD_W,
  maxY: WORLD_H
}, deltaMs)
// Then in Camera.ts clamping: needs cam.viewportW to be world-space viewport width.
// Easiest fix: set cam.viewportW = canvas.width / zoom, cam.viewportH = canvas.height / zoom
// in the ResizeObserver (or in update() each frame).
```

### Pattern 4: Wang Tile Index Decoding
**What:** Each tileset PNG is a 4×4 grid of 16 tiles (128×128px total, each tile 32×32px). The `transition-map.json` specifies per-cell edge transitions by tilesetId and direction. The Wang index (0..15) is a 4-bit bitmask encoding which of the 4 edges have the "upper" terrain.

**Wang standard index mapping (standard):**
```
Bit 0 = North edge is "upper" terrain
Bit 1 = East edge is "upper" terrain
Bit 2 = South edge is "upper" terrain
Bit 3 = West edge is "upper" terrain
```
Tile column = index % 4, tile row = floor(index / 4).

**But this phase has a simpler approach**: the transition-map.json already contains the final tilesetId per edge. The task is to determine the Wang index from the active edges for each cell and look up the correct tile in the corresponding 4×4 PNG.

**Recommended approach** — build a per-cell Wang index from transition-map entries:
```typescript
// For a cell at (x,y), check all 4 edges in the transition-map
// Each edge that transitions "to" the upper terrain contributes to the Wang index
// Edge-to-bit: north=0, east=1, south=2, west=3
// Wang index = bitmask of which edges border the upper terrain
```

### Pattern 5: Tile Coordinate to Pixel Conversion
**What:** terrain-map.json and transition-map.json use tile-space coordinates in range (−46..+54, −43..+57). Map these to pixel-space with an offset.

```typescript
// Tile (0,0) in tile-space maps to pixel (0,0) in world-space is WRONG.
// Tile-space origin (−46, −43) maps to pixel (0, 0):
const TILE_ORIGIN_X = -46   // from mapConfig.boundingBox.minX
const TILE_ORIGIN_Y = -43   // from mapConfig.boundingBox.minY
const TILE_SIZE = 32

function tileToPixel(tx: number, ty: number): { px: number; py: number } {
  return {
    px: (tx - TILE_ORIGIN_X) * TILE_SIZE,
    py: (ty - TILE_ORIGIN_Y) * TILE_SIZE,
  }
}
// tileToPixel(-46, -43) → (0, 0) ✓
// tileToPixel(54, 57)   → (3200, 3200) ✓  (100*32 = 3200, close to 3232)
```

### Anti-Patterns to Avoid
- **Drawing tiles per frame:** Never call `ctx.drawImage(tilesetImg, ...)` for each of 10k tiles inside `render()`. Pre-render to OffscreenCanvas once.
- **Scaling the OffscreenCanvas to 6464×6464:** Would require 160MB RAM for the offscreen buffer. Draw at world-space (3232×3232), scale at render time.
- **Hardcoding tile-coord origin as (0,0):** The map uses −46..+54 x range; pixel offset must be applied via tileToPixel().
- **Loading images synchronously:** All tileset and object PNGs must be loaded via `new Image()` with onload callbacks (or Promise wrappers) before drawing the offscreen canvas.
- **Setting `cam.viewportW` to screen pixels without zoom compensation:** Camera clamping will be wrong — the camera will stop too early and leave black borders. `viewportW` passed to updateCamera must be in world-space units (screen_px / zoom).

**Key insight:** The map export is self-contained and complete — every tile selection is pre-computed in transition-map.json. This phase is data-driven, not algorithmic.

## Common Pitfalls

### Pitfall 1: Tile Coordinate Offset
**What goes wrong:** Drawing terrain at tile (0,0) renders at pixel (0,0) when it should render at pixel (46*32, 43*32) = (1472, 1376).
**Why it happens:** The tile coordinate system uses a centered origin; world-space pixel origin is the top-left corner of the 3232×3232 canvas.
**How to avoid:** Apply `tileToPixel()` consistently using `TILE_ORIGIN_X = mapConfig.boundingBox.minX` and `TILE_ORIGIN_Y = mapConfig.boundingBox.minY`.
**Warning signs:** Map renders offset from center; objects appear misaligned with terrain.

### Pitfall 2: Camera Bounds After Zoom
**What goes wrong:** At 2x zoom, the effective visible area is half the screen dimensions in world-space. If `cam.viewportW` is set to the screen pixel width (e.g. 1280), the clamping formula `cam.x <= WORLD_W - cam.viewportW` → `cam.x <= 3232 - 1280 = 1952` is wrong. The correct clamp is `cam.x <= 3232 - 640 = 2592`.
**Why it happens:** `viewportW` in CameraState represents the world-space half of the viewport, not the screen size, when zoom > 1.
**How to avoid:** In the ResizeObserver (or each frame), set `cam.viewportW = canvas.width / zoom` and `cam.viewportH = canvas.height / zoom`. The `updateCamera()` function itself needs no changes.
**Warning signs:** Black space appears at bottom/right edges; or player walks off the visible area before camera stops.

### Pitfall 3: Image Loading Race
**What goes wrong:** OffscreenCanvas is drawn before tileset images finish loading → blank or partial map.
**Why it happens:** `new Image().src = ...` is async; the onload callback hasn't fired yet when rendering starts.
**How to avoid:** Wrap all image loads in `Promise`s; call `await Promise.all(imageLoads)` before drawing the offscreen canvas; set `this.ready = true` only after all images are loaded.
**Warning signs:** First frame shows black canvas or partial tiles.

### Pitfall 4: OffscreenCanvas in Tests
**What goes wrong:** `new OffscreenCanvas(...)` throws in jsdom (vitest environment) because jsdom doesn't implement it.
**Why it happens:** OffscreenCanvas is a browser-only API.
**How to avoid:** Add a jsdom stub in `setupTests.ts` (same pattern as existing HTMLCanvasElement stub):
```typescript
// setupTests.ts
if (typeof OffscreenCanvas === 'undefined') {
  global.OffscreenCanvas = class {
    constructor(w: number, h: number) { this.width = w; this.height = h }
    getContext() { return { drawImage: () => {}, fillRect: () => {}, /* ... */ } }
  } as unknown as typeof OffscreenCanvas
}
```
**Warning signs:** Vitest throws `ReferenceError: OffscreenCanvas is not defined`.

### Pitfall 5: Object Bounding Box Coordinates are World-Space, not Tile-Space
**What goes wrong:** Applying the tile-to-pixel offset to object coordinates produces wrong positions.
**Why it happens:** Objects in `manifest.json` have `boundingBox.{x,y}` in pixel world-space (same coordinate system as the 3232×3232 canvas), NOT in tile coordinates. The tile offset only applies to terrain-map and transition-map entries.
**How to avoid:** Draw objects directly at `boundingBox.x, boundingBox.y` with no further offset.
**Warning signs:** Objects render hundreds of pixels off from their intended positions.

### Pitfall 6: Default Terrain (terrainId: 0)
**What goes wrong:** Cells not listed in terrain-map.json have `defaultTerrain: 0`, but terrain ID 0 is not defined in map.json terrains array (IDs are 1–7). These cells would fail a lookup.
**Why it happens:** `defaultTerrain: 0` likely means "no terrain" / "use nothing" or the cell is not part of the map.
**How to avoid:** Treat `terrainId: 0` as transparent/skip. Only draw cells that appear in terrain-map.json. The 101×101 grid has 10,201 cells total but terrain-map is sparse — only listed cells have terrain.
**Warning signs:** Canvas errors attempting to draw with undefined tileset.

### Tileset PNG Slice (Wang tile lookup)
```typescript
// 4x4 grid, each tile 32x32px
// wangIndex: 0..15
function drawWangTile(
  ctx: OffscreenCanvasRenderingContext2D,
  tilesetImg: HTMLImageElement,
  wangIndex: number,
  destX: number,
  destY: number,
): void {
  const TILE_SIZE = 32
  const srcCol = wangIndex % 4
  const srcRow = Math.floor(wangIndex / 4)
  ctx.drawImage(
    tilesetImg,
    srcCol * TILE_SIZE, srcRow * TILE_SIZE, TILE_SIZE, TILE_SIZE,
    destX, destY, TILE_SIZE, TILE_SIZE,
  )
}
```

## Data Format Reference

### map.json — Key fields
- `mapConfig.tileSize`: 32
- `mapConfig.dimensions`: width=101, height=101, pixelWidth=3232, pixelHeight=3232
- `mapConfig.boundingBox`: minX=-46, minY=-43, maxX=54, maxY=57
- `terrains[]`: array of {id, name, color, baseTileId}; IDs 1–7; no ID 0
- `tilesets[]`: array of {id (UUID), filename, lowerTerrainId, upperTerrainId, gridLayout:"4x4", wangIndexMapping:"standard"}
- `overlays[]`: [{id:0, filename:"overlay-0.png", worldCoordinates:{x:128,y:96}, dimensions:{width:672,height:832}}]

### terrain-map.json — Key fields
- `format`: "sparse"
- `defaultTerrain`: 0 (means "no terrain" — cells not listed are undefined/transparent)
- `cells[]`: [{x, y, terrainId}] — tile-space coordinates

### transition-map.json — Key fields
- `format`: "sparse"
- `transitions[]`: [{x, y, edges: {north?, east?, south?, west?: {fromTerrain, toTerrain, tilesetId, transitionSize}}}]
- Coordinates are tile-space. Only cells with at least one transition edge appear.

### objects/manifest.json — Key fields
- `objects[]`: [{id, filename, boundingBox:{x,y,width,height}, layer, visible}]
- `boundingBox.x/y` are **pixel world-space** (NOT tile coords), range roughly 230–1100px x, 50–850px y
- `layer` is an integer (2–30); sort ascending for correct draw order
- Object PNG files: `{UUID}.png`, served from `/map/objects/{UUID}.png`

## Open Questions

1. **Wang index derivation from transition-map edges**
   - What we know: The transition-map gives per-cell edge transitions with tilesetId and direction. Standard Wang uses a 4-bit bitmask (N/E/S/W = bits 0/1/2/3 for the "upper" terrain).
   - What's unclear: The transition-map records the direction of the edge (e.g., "south" edge of cell transitions from terrain A to terrain B). Whether "south edge is upper terrain" means bit 2 is set depends on the Wang convention for this particular exporter.
   - Recommendation: For the first implementation, use the 4-bit bitmask from which edges in the cell have a transition entry. If visual artifacts appear, flip the bit convention. The tileset PNGs can be visually inspected to verify.

2. **Base terrain fill for non-transitioned cells**
   - What we know: terrain-map.json lists each cell's terrainId. The tilesets array in map.json includes a `baseTileId` per terrain, but this UUID is a terrain-level identifier, not a tileset UUID.
   - What's unclear: Which tile in which tileset to use for a "pure" cell with no transitions on any edge. The baseTileId doesn't directly map to a tileset PNG.
   - Recommendation: For pure cells (no edges in transition-map), use Wang index 0 (all-lower terrain tile) of the tileset that has `lowerTerrainId` matching the cell's terrain. If no tileset exists for same-terrain pairs, use a solid color fill matching `terrain.color` (#9ca3af for all 7 terrains in this export) as fallback.

3. **Only 2 tileset PNGs found vs 7 tilesets in map.json**
   - What we know: `ls tilesets/` shows only 2 PNG files present. map.json lists 7 tileset entries.
   - What's unclear: Whether the remaining 5 tileset PNGs simply weren't copied to the working directory or don't exist in the export.
   - Recommendation: Verify the full export directory. If only 2 PNGs are present, fallback to color fill (`#9ca3af`) for missing tilesets and log a warning.
