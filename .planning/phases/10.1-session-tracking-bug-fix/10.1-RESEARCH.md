# Phase 10.1: Session Tracking Bug Fix - Research

**Researched:** 2026-04-08
**Domain:** SQLite persistent session mapping, daemon restart recovery, session lifecycle management
**Confidence:** HIGH

## Summary

Phase 10.1 addresses a critical bug where Claude Code sessions become invisible in the UI after daemon restarts. The root cause is the in-memory `sessionIdCache` in `hookParser.ts` that maps Claude's `session_id` to Cockpit's UUID. When the daemon restarts, this cache is lost, causing existing active Claude sessions to receive new UUIDs and emit events without `session_start` events. Since the UI's `sessionsSlice` only creates session records on `session_start` events, these sessions never appear in the session list.

The solution requires persisting the Claude session ID → Cockpit UUID mapping in SQLite, following the existing pattern used by the Codex adapter (`codex_sessions` table). A new `claude_sessions` table will store this mapping, and `hookParser.ts` will be refactored to query this table before generating new UUIDs. The daemon initialization will load the cache from the database on startup, ensuring session continuity across restarts.

**Primary recommendation:** Create a `claude_sessions` table mirroring `codex_sessions`, refactor `getOrCreateSessionId()` to use SQLite persistence, and implement a startup cache population in the daemon's initialization sequence.

### Pattern 1: Persistent Session Mapping Table
**What:** A SQLite table that maps external provider session IDs to Cockpit's internal UUIDs, surviving daemon restarts.
**When to use:** When you need to maintain identity continuity across process boundaries for provider-specific session identifiers.
**Example:**
```sql
-- Source: Existing codex_sessions table pattern in database.ts
CREATE TABLE IF NOT EXISTS claude_sessions (
  session_id   TEXT PRIMARY KEY,   -- Cockpit's UUID (our internal session identifier)
  claude_id    TEXT NOT NULL,      -- Claude's session_id from hook payload
  workspace    TEXT NOT NULL,      -- workspace path for context
  created_at   TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_claude_sessions_claude_id
  ON claude_sessions (claude_id);
```

**Rationale:** Mirrors the existing `codex_sessions` table structure, maintaining consistency across provider adapters. The `claude_id` index enables fast lookups when processing hook events. `created_at` provides useful metadata for debugging and potential cleanup policies.

### Pattern 2: Cache Initialization on Daemon Startup
**What:** Populate an in-memory cache from the persistent table at daemon startup to avoid database queries during hot event processing paths.
**When to use:** When you have high-throughput operations (hook processing) that need low-latency lookups, but data persistence is required for restart recovery.
**Example:**
```typescript
// Source: Based on daemon initialization pattern in index.ts
const db = openDatabase(DB_PATH);

// Initialize Claude session cache after DB opens
const claudeSessionCache = new Map<string, string>();
const rows = db.prepare('SELECT claude_id, session_id FROM claude_sessions').all() as Array<{
  claude_id: string;
  session_id: string;
}>;
for (const row of rows) {
  claudeSessionCache.set(row.claude_id, row.session_id);
}

// Hook server receives the cache
const hookServer = createHookServer(HOOK_PORT, claudeSessionCache, db, onEvent, onDecisionNeeded);
```

**Rationale:** The cache provides O(1) lookup performance during hook processing, while the database provides persistence. The Map is populated once at startup, avoiding per-event database queries that could become a bottleneck under high load.

### Pattern 3: Lazy Cache Population with Database Fallback
**What:** Check in-memory cache first, then fall back to database query if not found, populating cache for future lookups.
**When to use:** When you want the performance of in-memory caching but also need to handle late-arriving sessions that weren't cached at startup.
**Example:**
```typescript
// Source: Refactored getOrCreateSessionId pattern
function getOrCreateSessionId(
  claudeSessionId: string,
  db: Database.Database,
  cache: Map<string, string>,
): string {
  // Fast path: cache hit
  let uuid = cache.get(claudeSessionId);
  if (uuid) {
    return uuid;
  }

  // Medium path: database hit (e.g., session started before daemon restart)
  const row = db.prepare('SELECT session_id FROM claude_sessions WHERE claude_id = ?')
    .get(claudeSessionId) as { session_id: string } | undefined;
  if (row) {
    uuid = row.session_id;
    cache.set(claudeSessionId, uuid);
    return uuid;
  }

  // Slow path: create new UUID and persist
  uuid = randomUUID();
  db.prepare('INSERT INTO claude_sessions (session_id, claude_id, workspace, created_at) VALUES (?, ?, ?, ?)')
    .run(uuid, claudeSessionId, workspace, new Date().toISOString());
  cache.set(claudeSessionId, uuid);
  return uuid;
}
```

**Rationale:** Three-tier strategy optimizes for the common case (cache hit), handles restart recovery (database hit), and creates new sessions when needed. The database is the single source of truth; the cache is just a performance optimization.

### Anti-Patterns to Avoid
- **Database query for every hook event:** Even with SQLite's speed, querying the database for each hook payload would add unnecessary latency. Use in-memory cache with lazy population instead.
- **Cache without persistence:** The current bug exists because the cache is not persisted. Always maintain the database as the source of truth.
- **Synchronous cache population in hot path:** Loading all session mappings at startup is fine, but don't do it mid-event processing. Initialize the cache during daemon boot, before accepting hook connections.
- **Missing workspace path in table:** The workspace path is needed for `session_start` events. Don't omit it from the schema just because it's not strictly needed for the mapping itself.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session ID cache | Custom Map implementation with manual persistence | `Map<string, string>` + SQLite table | Built-in Map provides O(1) lookup; SQLite handles persistence, concurrency, and crash recovery |
| UUID generation | Custom ID generation logic | Node.js `randomUUID()` | Cryptographically secure, collision-resistant, already used in the codebase |
| Database migration scripts | Complex versioned migration system | `CREATE TABLE IF NOT EXISTS` + `INSERT OR IGNORE` | Idempotent schema evolution without migration version tracking; matches Phase 8 FTS5 pattern |

**Key insight:** The problem is simple key-value persistence. SQLite + in-memory cache is the right tool. Don't introduce complexity with versioned migrations or external cache systems.

## Common Pitfalls

### Pitfall 1: Race Condition Between Cache Load and First Hook
**What goes wrong:** A hook event arrives before the daemon finishes loading the cache from the database, causing a duplicate session_id to be created.
**Why it happens:** The hook server starts listening before cache initialization completes.
**How to avoid:** Initialize the cache synchronously in `index.ts` before calling `createHookServer()`. The `openDatabase()` call is synchronous; the cache population should also be synchronous.
**Warning signs:** Seeing duplicate `claude_id` entries in the database or multiple `session_id` values for the same `claude_id`.

### Pitfall 2: Missing `session_start` Emission for Resumed Sessions
**What goes wrong:** Daemon restarts, cache loads correctly, but existing sessions still don't appear in the UI because no `session_start` event is emitted.
**Why it happens:** The code only emits `session_start` when processing a `SessionStart` hook event. If the daemon restarts after that event was already processed, it won't be emitted again.
**How to avoid:** This is actually **correct behavior**. The UI's WebSocket catch-up protocol (`lastSeenSequence=0`) replays all historical events, including the original `session_start`. The UI will receive the original event during replay and create the session record correctly.
**Warning signs:** Sessions still invisible after daemon restart even though the mapping table has the correct data. Check that the UI is connecting with `lastSeenSequence=0` (default) and that `session_start` events exist in the events table.

### Pitfall 3: Workspace Path Not Available on Cache Hit
**What goes wrong:** When loading from cache/database, the `workspace` field isn't available for the `session_start` event.
**Why it happens:** The original `SessionStart` hook included `cwd`, but subsequent events (e.g., `PreToolUse`) don't include it. If we only cache the UUID, we lose the workspace path.
**How to avoid:** Store `workspace` in the `claude_sessions` table and use it when needed. For the bug fix, this isn't strictly necessary because `session_start` is only emitted on actual `SessionStart` hooks (which include `cwd`), not on cache hits. But including it in the table provides useful metadata and enables future features.
**Warning signs:** `session_start` events with empty `workspacePath` fields after daemon restart.

### Pitfall 4: Database Contention During High Hook Volume
**What goes wrong:** Under heavy load, SQLite database locks cause hook processing to stall or timeout.
**Why it happens:** Multiple concurrent writes to the `claude_sessions` table can cause WAL lock contention.
**How to avoid:** The three-tier cache strategy (cache → DB → create) minimizes database writes: writes only happen for new sessions, not for existing ones. SQLite's WAL mode (already enabled in the project) handles concurrent reads efficiently. For extreme load, consider batching writes, but this is premature optimization for the current use case.
**Warning signs:** Hook latency increasing under load, `database is locked` errors in logs.

### Pitfall 5: Stale Cache After Database Changes
**What goes wrong:** External process modifies the `claude_sessions` table, but the in-memory cache doesn't reflect the changes.
**Why it happens:** The cache is only populated at startup; it doesn't watch for external changes.
**How to avoid:** For this use case, it's acceptable. The daemon is the sole writer to `claude_sessions`, and external readers (e.g., manual SQL queries) don't need real-time consistency. If this becomes a requirement, implement cache invalidation or reload on a signal.
**Warning signs:** Discrepancies between `SELECT * FROM claude_sessions` and what the daemon believes the current sessions are.

## Open Questions

1. **Should we retroactively emit `session_start` events for orphaned sessions?**
   - What we know: Events exist in the database for sessions without corresponding `session_start` events
   - What's unclear: Whether the UI can handle late-arriving `session_start` events after other events have already been processed
   - Recommendation: **Do not retroactively emit**. The WebSocket catch-up protocol replays events in sequence. If we insert a `session_start` event with an old timestamp, it would appear out of sequence. Instead, rely on the existing `session_start` events already in the database (from when the sessions originally started). The bug only affects sessions active during daemon restart, not historical sessions.

2. **What happens when a Claude session ends and the daemon restarts?**
   - What we know: The `claude_sessions` row persists after session end (no `ON DELETE CASCADE`)
   - What's unclear: Whether stale rows should be cleaned up
   - Recommendation: Keep rows indefinitely. They provide useful audit trail and don't consume significant storage. If cleanup is needed later, add a cleanup job that removes rows older than X days where the session has ended (check events table for `session_end` event).

3. **How do we handle the case where the same Claude session_id is reused?**
   - What we know: Claude's `session_id` is unique per session (based on current observations)
   - What's unclear: Whether Claude ever recycles session IDs across different sessions
   - Recommendation: Treat Claude `session_id` as unique and never reused. If evidence of reuse emerges, add a `UNIQUE` constraint on `(claude_id, created_at)` or add a `last_seen_at` timestamp and update it on each event, allowing session_id reuse with temporal separation. For now, this is premature optimization.
