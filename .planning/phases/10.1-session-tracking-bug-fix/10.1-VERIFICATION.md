### Human Verification Required

#### 1. Daemon Restart Preserves Session UUID

**Test:** Start Claude Code with hooks configured to point at the Cockpit daemon. Note the session UUID that appears in the UI. Stop the daemon (`SIGTERM`). Restart the daemon. Send a hook event from the same running Claude Code session (e.g., trigger a tool use).
**Expected:** The event is attributed to the original session UUID, not a new one. The session remains visible in the UI session list.
**Why human:** Requires a live daemon process, a running Claude Code session with hooks wired, and observation across a daemon restart boundary. Unit tests inject an in-memory DB and verify the three-tier lookup logic, but cannot simulate the full OS process lifecycle.

#### 2. Sessions Remain Visible in UI After Daemon Restart

**Test:** Open the Cockpit UI. Confirm at least one active session appears in the session list. Stop and restart the daemon. Check that the session list still shows the same session(s) with the same names/identifiers.
**Expected:** No sessions disappear from the UI after restart. If the Claude session is still active, its events continue to arrive under the correct session record.
**Why human:** Depends on WebSocket reconnection, daemon startup timing, cache hydration latency, and frontend session state ã none of which are exercised by unit tests.

### Summary

Phase 10.1 goal is fully achieved at the implementation and unit-test level. The core bug (in-memory-only `sessionIdCache` lost on restart) is fixed by:

1. A persistent `claude_sessions` SQLite table with `UNIQUE` constraint on `claude_id`
2. `initializeClaudeSessionCache()` hydrating the in-memory cache from the table at startup
3. `getOrCreateSessionId()` using three-tier lookup (cache ã DB ã create) so a restarted daemon recovers existing mappings
4. Correct startup ordering in `index.ts` ensuring cache is populated before the hook server accepts connections

Two items require human verification: the live daemon restart scenario and end-to-end UI visibility. These cannot be automated without an integration test harness spanning the full daemon process lifecycle.
