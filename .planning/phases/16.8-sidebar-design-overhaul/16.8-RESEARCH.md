## Summary

You can plan this phase well if you treat it as both a **visual redesign** and a **UI infrastructure correction**.

The current sidebar has four concrete technical issues that directly affect planning:

1. Semantic Tailwind classes used by the sidebar (`bg-sidebar`, `border-border`, `text-muted-foreground`, `bg-background`, etc.) are not present in generated CSS, so styling tokens are effectively missing right now.
2. Layout is structurally awkward: `OpsLayout` renders an outer fixed sidebar (`w-72`) and `MapSidebar` renders another nested `<aside>` (`w-56`) with its own border/background, creating mismatched widths and double-surface behavior.
3. Session row behavior is partially inconsistent with current store types: `MapSidebar` color map includes `failed`, but `SessionStatus` is `active | ended | error`.
4. Existing tests for `MapSidebar` are weak/incorrect (empty state test does not actually verify empty state), so visual/behavior regressions can slip through.

Also note: `onFocusSession` wiring exists, but `scrollToSession` in `OfficePage.tsx` is currently a no-op placeholder. Since context says sidebar click should still focus map session, planning should explicitly decide whether to restore that behavior in this phase or track as a dependency/follow-up.

## Architecture Patterns

### Pattern 1: Single Sidebar Surface Owned by `OpsLayout`

**What:** Keep `OpsLayout` as container owner; make `MapSidebar` content-only (no nested width/border-owning `<aside>`).

**Why:** Removes double-sidebar geometry and makes width-resize logic unambiguous.

**Recommended structure:**
- `OpsLayout.tsx`: owns width state, header, resize handle, container surface
- `MapSidebar.tsx`: owns list rendering + row interactions + empty state

### Pattern 2: Desktop Resizer with Persisted Width

**What:** Use pointer events and persisted width (`useLocalStorage`) in `OpsLayout`.

**Why:** Matches context decision (user-resizable on desktop) without new library.

**Rules to lock:**
- Clamp width between min/max
- Disable drag handle on small screens
- Persist width key (e.g. `cockpit.sidebar.width`)

### Pattern 3: Sidebar Row View-Model

**What:** Derive a sidebar-specific row model from `useActiveSessions()` in `MapSidebar`, sort by recency (`lastEventAt` desc), and normalize display labels.

**Why:** Keeps sorting/display concerns local; avoids unintended behavior change in other consumers of `useActiveSessions`.

### Pattern 4: Visual Hierarchy Contract

Use three clear levels:
- **Header block:** title + subtle subtitle/utility count
- **Primary list:** interactive rows with strong selected/focused state
- **Secondary metadata:** provider/status text, approval badge, optional second line

### Pattern 5: Status Semantics as Dot + Text

**What:** Always pair color with text label (status chip/text).

**Why:** Context requires non-color-only status; improves accessibility and scan speed.

Also correct status map to include `error` (not `failed`) for type compatibility.

### Pattern 6: Regression-Proof Testing

Split tests into:
- `OpsLayout` shell + resize behavior tests
- `MapSidebar` row rendering, recency sort, empty state, click callback, badge/status/provider display

Rewrite the current empty-state test (it is currently not asserting the intended behavior).

## Common Pitfalls

### Pitfall 1: Missing semantic token definitions

**What goes wrong:** Classes like `bg-sidebar`/`text-foreground` compile to nothing, leaving visuals broken.

**How to avoid:** Define semantic color variables in Tailwind v4 theme scope before redesign styling work.

### Pitfall 2: Nested width ownership

**What goes wrong:** `OpsLayout` and `MapSidebar` both manage width/surface styles, causing inconsistent spacing and border artifacts.

**How to avoid:** One owner (`OpsLayout`) for sidebar geometry/surface.

### Pitfall 3: Status enum mismatch

**What goes wrong:** UI color map expects `failed`, store emits `error`.

**How to avoid:** Align to `SessionStatus` type and include text labels.

### Pitfall 4: Recency ordering omitted

**What goes wrong:** Sessions render in object insertion order rather than most-recent-first.

**How to avoid:** Explicit `lastEventAt` sort in sidebar row model.

### Pitfall 5: “Focus session” assumption drift

**What goes wrong:** Design phase ships polished rows but row click still does not focus map due no-op callback.

**How to avoid:** Make focus behavior an explicit acceptance criterion or explicit deferred item.

### Pitfall 6: Test suite gives false confidence

**What goes wrong:** Existing tests pass while missing real empty-state and interaction assertions.

**How to avoid:** Replace brittle mocks with explicit assertion cases tied to context decisions.

### 2) Desktop resizable sidebar owner pattern

```tsx
// OpsLayout.tsx (shape only)
const [sidebarW, setSidebarW] = useLocalStorage('cockpit.sidebar.width', 320)
const minW = 260
const maxW = 460

function startResize(e: React.PointerEvent<HTMLDivElement>) {
  const startX = e.clientX
  const startW = sidebarW
  function onMove(ev: PointerEvent) {
    setSidebarW(Math.max(minW, Math.min(maxW, startW + (ev.clientX - startX))))
  }
  function onUp() {
    window.removeEventListener('pointermove', onMove)
    window.removeEventListener('pointerup', onUp)
  }
  window.addEventListener('pointermove', onMove)
  window.addEventListener('pointerup', onUp)
}
```

### 3) Sidebar-local row model and sorting

```ts
const sessions = useActiveSessions()

const rows = [...sessions]
  .sort((a, b) => b.lastEventAt.localeCompare(a.lastEventAt))
  .map((s) => ({
    id: s.sessionId,
    project: s.workspacePath.split('/').at(-1) ?? s.sessionId,
    provider: s.provider,
    status: s.status, // active | ended | error
    pendingApprovals: s.pendingApprovals,
  }))
```
