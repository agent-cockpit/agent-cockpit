**Gathered:** 2026-04-13

<domain>
## Phase Boundary

Redesign the existing Office/Ops sidebar presentation to be visually polished and coherent with the space/pixel-art game theme. This phase improves layout, typography, spacing, hierarchy, and interaction states for the current sidebar behavior (active session navigation + map focus), without adding new product capabilities.

</domain>

<decisions>
## Implementation Decisions

### Scope Guardrails Carried Forward
- Keep Phase 14 sidebar role: active-agent navigation remains the core behavior.
- Clicking a session row still focuses that session on the map (existing behavior preserved).
- Redesign focuses on presentation quality and interaction polish, not feature expansion.

### Information Density
- Each row shows: project name, status, and provider.
- Rows use a second line only when needed (conditional metadata), not always-on subtext.
- Pending approvals are shown as a numeric pill badge.
- Session ordering is by most recent activity first.

### Visual Direction
- Style direction: hybrid sci-fi + pixel-art aesthetic.
- Palette anchor: slate base with cyan + amber accents.
- Typography: pixel flavor for labels/display moments, readability-first body text for list rows.
- Surface treatment: layered panel styling with subtle glow accents.

### Layout Structure
- Sidebar width is user-resizable on desktop.
- Top region should be a single unified header block (instead of split bars).
- Content organization stays focused on one primary active-session list section.
- Empty state should be themed but minimal.

### Interaction and State Cues
- Focused row uses a strong persistent active highlight.
- Hover behavior uses subtle lift + color shift.
- Status uses dot + text state label (not color-only).
- Pending-approval urgency uses numeric badge plus restrained accent ring emphasis.

### Claude's Discretion
- Exact spacing scale, corner radii, border thickness, and glow intensity.
- Exact iconography choices and status label wording.
- Animation timing/easing values for hover/active transitions.
- Breakpoint details for width behavior and resizer affordance implementation.

</decisions>

## Existing Code Insights

### Reusable Assets
- `packages/ui/src/components/layout/MapSidebar.tsx`: current sidebar skeleton, row mapping, and status-dot pattern.
- `packages/ui/src/components/layout/OpsLayout.tsx`: host shell where sidebar header composition and width behavior are controlled.
- `packages/ui/src/store/selectors.ts` (`useActiveSessions`): active-session source for sidebar rows.
- `packages/ui/src/store/index.ts` (`SessionRecord`): available row metadata fields (`provider`, `status`, `pendingApprovals`, `lastEventAt`, `workspacePath`).
- `packages/ui/src/components/office/InstancePopupHub.tsx` and `packages/ui/src/components/layout/SessionDetailPanel.tsx`: existing panel/tab visual patterns that can inform hierarchy consistency.

### Established Patterns
- Tailwind utility-class styling is used across layout/panel components.
- Sidebar currently uses tokenized classes (`bg-sidebar`, `border-border`, `text-muted-foreground`) for theme-level consistency.
- Active sessions are currently filtered in selector logic, then rendered as simple button rows.
- Map focus behavior is already wired through `onFocusSession` and `scrollToSession`.

### Integration Points
- Main redesign implementation lives in `MapSidebar.tsx` (rows, hierarchy, states, badges, empty state).
- Header/structure and width behavior updates integrate in `OpsLayout.tsx`.
- Behavior tests should extend `packages/ui/src/components/layout/__tests__/MapSidebar.test.tsx`.

</code_context>

<specifics>
## Specific Ideas

- Keep the sidebar readable first, but make it feel like a cockpit UI instead of generic app chrome.
- Use emphasis hierarchy so action-needed sessions are obvious at a glance without visual overload.
- Preserve compatibility with current map-flow interactions while upgrading presentation quality.

</specifics>
