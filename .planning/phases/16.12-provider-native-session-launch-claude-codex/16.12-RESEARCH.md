## Planning TL;DR

This phase is mostly a daemon launch-contract correction plus modal state correction.  
The current Claude launch path is outdated/broken, Codex readiness is optimistic in the UI, and Claude launch/session ID mapping is currently inconsistent with strict event-driven readiness.

Before writing plans, lock three decisions:
1. **Claude launch mode** (headless mode that can run from daemon without manual terminal copy).
2. **Claude session ID alignment strategy** (launched session ID must equal hook-derived session ID).
3. **Launch readiness contract** (UI waits for `session_start`, not POST success).

## Critical Findings (What The Planner Must Know)

### 1) Current Claude launch command is invalid on installed Claude CLI

Current daemon code builds:
`COCKPIT_SESSION_ID=<uuid> claude --hooks http://localhost:<port>/hook`

But on the installed CLI (`Claude Code 2.1.105`), `--hooks` is not a valid flag:

```bash
$ claude --hooks http://localhost:3002/hook -p "ping"
error: unknown option '--hooks'
(Did you mean --tools?)
```

Impact: current "configure-and-copy" command path is functionally broken for modern Claude CLI.

### 2) Claude launch `sessionId` returned by API is not guaranteed to be the real session ID

- `handleLaunchSession()` generates a UUID and returns it for Claude.
- Claude hook parser currently maps `payload.session_id` to a UUID via cache/DB lookup and may create a new UUID.
- `COCKPIT_SESSION_ID` env var is currently not consumed anywhere in daemon hook parsing.

Impact: launch response ID and actual tracked session can diverge, breaking event-driven readiness checks.

### 3) UI currently treats launch as success too early

- Modal shows codex success immediately after POST 200.
- Phase decision requires readiness only after real `session_start`.

Impact: modal state machine must add "launch requested -> waiting for `session_start` -> ready/error".

### 4) Launch errors are under-modeled in UI session state

- Session records are created only on `session_start`.
- `provider_parse_error` for unknown session does not create/update a session record.

Impact: without explicit launch error path, failures can be silent from user perspective.

### 5) Project-specific instruction checks

- `./CLAUDE.md`: not present.
- `.claude/skills/`: not present.
- `.agents/skills/`: not present.

Impact: no extra project-local skill conventions to incorporate in this phase plan.

- `packages/daemon/src/ws/server.ts`
  - `handleLaunchSession()` branches by provider.
  - Claude path: returns manual command (broken flag).
  - Codex path: instantiates `CodexAdapter` and returns success immediately.

### Codex launch/session lifecycle

- `packages/daemon/src/adapters/codex/codexAdapter.ts`
  - Spawns `codex app-server`.
  - Performs `initialize` + `thread/start|resume`.
  - Emits `session_start` after successful thread bootstrap.

### Claude lifecycle ingestion

- `packages/daemon/src/adapters/claude/hookServer.ts`
  - Receives hook HTTP POST.
  - Emits synthetic `session_start` on first non-start event when needed.

- `packages/daemon/src/adapters/claude/hookParser.ts`
  - Maps Claude `session_id` -> cockpit UUID via cache/DB/random UUID creation.

- `packages/ui/src/components/sessions/LaunchSessionModal.tsx`
  - Assumes Claude returns `hookCommand` and shows copy UI.

- `packages/ui/src/store/sessionsSlice.ts`
  - Session exists only after `session_start`.
  - No "launch pending" state in store (modal-local waiting state needed).

## Design Implications For Planning

## 1) Claude launcher design is the main planning fork

A strict-auto launch with minimal input must run without manual terminal copy.

Two realistic approaches:

1. **Headless Claude mode from daemon** (preferred if validated):
   - Use a Claude mode that can run without manual TTY interaction.
   - Inject HTTP hook configuration via `--settings` (or settings file path).
   - Pass deterministic `--session-id`.

2. **Platform terminal auto-open** (fallback design only if headless is not viable):
   - Daemon opens a terminal process and launches interactive Claude there.
   - Higher platform coupling and operational complexity.

Planning note: include an explicit "launch mode validation spike" as first task in first plan.

## 2) Session ID integrity must be solved explicitly

Best planning target:
- Launch Claude with deterministic `--session-id <uuid>`.
- Ensure hook parser preserves UUID IDs instead of remapping to a random UUID.

Reason: modal waiting logic can then key directly on launch response `sessionId`.

## 3) Hook wiring must move from deprecated command flag to settings-based HTTP hooks

Plan should include:
- Build hook config payload for relevant events (`SessionStart`, `SessionEnd`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `SubagentStart`, `SubagentStop`, `Notification`).
- Point each to daemon hook endpoint (`http://localhost:${HOOK_PORT}/hook`).
- Keep strict-auto behavior: configuration failure is launch failure.

## 4) Readiness contract must be event-based end-to-end

Launch response should mean "launch initiated", not "session ready".

UI modal states should be:
- `loading` (HTTP request),
- `waiting_for_session_start` (launch accepted),
- `ready` (session appears via `session_start`),
- `error` (preflight/spawn/timeout failure).

## 5) Launch error path must be explicit and user-actionable

Daemon should fail fast with structured errors for:
- invalid workspace path,
- missing provider binary,
- Claude not authenticated / unsupported launch mode preconditions,
- spawn failure.

No manual command fallback UX in this phase (per context decision).

## Suggested Plan Decomposition (Plan-Ready)

1. **Daemon launch contract + preflight**
   - Replace Claude configure-and-copy response with strict auto launch path.
   - Add per-provider preflight checks and structured error responses.
   - Keep Codex launch path but align response semantics to "initiated".

2. **Claude launcher implementation**
   - Implement launcher module with deterministic `sessionId`.
   - Configure hooks via settings-based HTTP hook definitions.
   - Guarantee process lifecycle handling and spawn error surfacing.

3. **Session ID mapping hardening**
   - Update hook session ID mapping logic so deterministic launch IDs remain stable.
   - Add tests for UUID passthrough and non-UUID compatibility.

4. **UI launch modal state machine**
   - Remove copy-command success path.
   - Add waiting-for-`session_start` UX with timeout + clear failure messaging.
   - Keep minimal inputs (workspace only).

5. **Tests + regression validation**
   - Update daemon launch tests (remove `configure-and-copy` expectations).
   - Add daemon tests for preflight failure matrix.
   - Add/extend UI tests for waiting and error states.

## Risks And Pitfalls To Plan Around

1. **Headless Claude runtime mode uncertainty**
   - Needs a short spike before coding broad changes.
   - Do not assume old `--hooks` workflow still applies.

2. **Backfill side effects if pre-registering Claude sessions**
   - `backfillSessionStarts()` inserts start events for claude session rows missing starts.
   - If rows are pre-created before successful launch, restart behavior can create phantom sessions.

3. **Silent launch failures**
   - If failure is only async and not propagated, UI can stall in waiting state.
   - Plan requires explicit timeout/error transition.

4. **Cross-provider parity drift**
   - Codex and Claude should share same launch API semantics (`initiated` vs `ready`).
