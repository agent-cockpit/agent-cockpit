# Phase 16.12: Provider-Native Session Launch (Claude + Codex) - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Enable launching new Claude and Codex sessions directly from the app's Launch Session flow, with provider-native runtime invocation from the daemon. This phase covers launch orchestration, launch-state reporting, and launch error handling for browser-initiated sessions. It does not replace Claude/Codex runtimes or add advanced orchestration features.

</domain>

<decisions>
## Implementation Decisions

### Launch Model
- Default behavior is strict automatic launch for both providers.
- If automatic launch is unavailable for Claude on a machine, treat it as a launch error (no manual command fallback UX in this phase).
- App should continue to use provider-native runtimes (Claude CLI and Codex app-server), not a custom runtime.

### Launch Inputs and Session Readiness
- Launch inputs remain minimal for this phase: workspace path only.
- Session should be marked ready/active only after receiving real `session_start` lifecycle events, not optimistic API success.

### Claude's Discretion
- Exact daemon-side launcher structure (service/module boundaries).
- Exact preflight checks and diagnostic detail format for launch failures.
- Exact error messaging style in modal while keeping strict-auto semantics.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/ui/src/components/sessions/LaunchSessionModal.tsx`: existing provider/workspace launch UX, loading/success/error states.
- `packages/daemon/src/ws/server.ts` (`handleLaunchSession`): current launch API endpoint and provider branching.
- `packages/daemon/src/adapters/codex/codexAdapter.ts`: existing spawned-process lifecycle, thread start/resume, and early `session_start` emission.
- `packages/daemon/src/adapters/claude/hookServer.ts`: existing lifecycle ingestion path that drives `session_start` for Claude sessions.

### Established Patterns
- Session visibility in UI is lifecycle-event-driven (`session_start`), not purely request-response driven.
- Launch requests currently flow through `POST /api/sessions` with provider + workspace payload.
- Provider-specific behavior is centralized in daemon API routing, then surfaced to UI through normalized events.

### Integration Points
- Extend daemon launch path in `packages/daemon/src/ws/server.ts` to support automatic Claude launch behavior.
- Keep Codex launch path aligned with current `CodexAdapter` behavior and error semantics.
- Update `LaunchSessionModal` success/error branches to match strict-auto/no-manual-fallback behavior.
- Ensure session list/map rendering remains bound to emitted `session_start` events from provider adapters.

</code_context>
