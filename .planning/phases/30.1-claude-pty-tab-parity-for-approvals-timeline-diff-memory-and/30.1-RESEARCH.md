## Planning TL;DR

The PTY work introduced a second transport path:

- raw terminal streaming over `pty_output`
- existing structured state over normalized events + REST hydration

The terminal path is working, but the other tabs are still built on the older structured path. The safest plan is:

1. Lock the PTY Claude parity contract with RED tests
2. Repair daemon/runtime structured event and session metadata flow
3. Repair or extend UI hydration paths only where parity still breaks

Do not solve this by scraping ANSI terminal output inside the UI.

## Critical Findings

### 1) PTY output is intentionally separate from normalized events

- `packages/ui/src/hooks/useSessionEvents.ts` routes `pty_output` directly into `ptyBus`
- `ApprovalInbox`, `TimelinePanel`, `DiffPanel`, `MemoryPanel`, and `ArtifactsPanel` do not consume `pty_output`

Impact: these tabs only work if structured events and session metadata remain correct under PTY Claude.

### 2) PTY Claude still depends on hook relay for structured events

- `packages/daemon/src/adapters/claude/ptyLauncher.ts` writes Claude hook settings and launches Claude in a PTY
- `packages/daemon/src/adapters/claude/hookServer.ts` remains the structured event ingestion path

Impact: if parity broke after PTY migration, the likely fault is not in the panels themselves first; it is in hook relay, session/workspace registration, or replay/hydration assumptions.

### 3) Popup mode switch only changes the chat surface

- `InstancePopupHub.tsx` swaps `ChatPanel` to `TerminalPanel` for `liveSession.mode === 'pty'`
- The other five tabs are unchanged and continue to rely on the store and REST endpoints

Impact: UI redesign is not needed. This is mainly a data-contract and hydration correctness phase.

### 4) Replay correctness is a distinct risk from live correctness

- `useSessionEvents.ts` buffers replay until `catchup_complete`
- several tabs also fetch REST hydration when local state is empty

Impact: a PTY Claude session can appear fine live but still fail after popup reopen or browser reconnect if structured events, session metadata, or empty-state fetch paths diverge.

### 5) Diff already has a tool-call fallback path

- `DiffPanel.tsx` can synthesize entries from `tool_call` events (`Write`, `Edit`, `Update`, `MultiEdit`) when `file_change` is missing

Impact: planning should verify whether daemon parity needs missing `file_change` fixed at source or whether existing tool metadata is sufficient for some cases. The answer may differ between `diff` and `artifacts`.

### 6) Memory depends on workspace linkage, not just event presence

- `MemoryPanel.tsx` loads `CLAUDE.md`, auto-memory, and notes through daemon endpoints keyed by `sessionId`
- daemon endpoints resolve workspace via `getWorkspacePath(db, sessionId)`

Impact: if PTY sessions do not persist the right session/workspace association early enough, memory can fail even when the terminal works.

## Current Architecture Relevant To 30.1

### PTY launch/runtime

- `packages/daemon/src/adapters/claude/ptyLauncher.ts`
  - configures Claude hooks through a generated settings file
  - persists Claude session ID mapping
  - streams raw PTY output and lifecycle events

- `packages/daemon/src/ws/server.ts`
  - launches PTY Claude sessions
  - maintains runtime registries
  - exposes memory/session REST endpoints used by the tabs

### Structured event path

- `packages/daemon/src/adapters/claude/hookServer.ts`
  - parses hook payloads
  - synthesizes `session_start` when needed
  - resolves pending approvals

- `packages/daemon/src/adapters/claude/hookParser.ts`
  - maps hook payloads into normalized events such as `approval_request`, `file_change`, `memory_read`, `memory_write`

- `packages/shared/src/events.ts`
  - defines the typed normalized event contract used by the UI

### UI/store consumers

- `packages/ui/src/hooks/useSessionEvents.ts`
  - catch-up buffering and batch apply
  - PTY side-channel routing

- `packages/ui/src/components/panels/ApprovalInbox.tsx`
  - store-based queue plus REST hydration for pending approvals

- `packages/ui/src/components/panels/TimelinePanel.tsx`
  - store event rendering plus empty-state fetch

- `packages/ui/src/components/panels/DiffPanel.tsx`
  - file-change view with tool-call fallback and empty-state fetch

- `packages/ui/src/components/panels/MemoryPanel.tsx`
  - event-backed suggestions plus workspace-backed HTTP reads/writes

- `packages/ui/src/components/panels/ArtifactsPanel.tsx`
  - derives visible artifacts from normalized events

## Design Implications For Planning

### 1) Contract tests must come first

There is not enough confidence yet about the exact failure point. A first plan should create RED tests around:

- PTY Claude launch/workspace/session metadata
- hook-derived structured events after PTY launch
- replay/catch-up behavior after reconnect
- empty-state tab hydration for PTY Claude sessions

Without this, execution risks patching the wrong layer.

### 2) Daemon/runtime is the likely highest-value fix layer

If structured events or workspace/session linkage are repaired at the source, most tab components should not need major changes. Planning should therefore bias implementation work toward daemon/runtime before UI.

### 3) UI work should be surgical and regression-oriented

Expected UI changes are likely to be:

- better hydration for PTY-mode sessions when local slices are empty
- regression tests for popup reopen and reconnect
- minor panel/store fixes where PTY mode exposed hidden assumptions

Avoid a broad panel rewrite.

### 4) Artifacts may need explicit parity decisions

`DiffPanel` already falls back to tool metadata. `ArtifactsPanel` may or may not need the same strategy depending on how it currently derives entries. This should be a planned verification point.

### 5) Memory parity must include both events and HTTP-backed content

A PTY Claude session is not “memory-correct” just because `memory_write` events exist. Planning should explicitly test:

- `CLAUDE.md` lookup
- auto-memory lookup
- notes fetch/write/delete
- suggested memory writes approval/rejection

   - define the expected source-of-truth and replay behavior

2. **Plan 02: Daemon/runtime parity fixes**

3. **Plan 03: Non-terminal UI tab parity**
   - restore any remaining panel hydration gaps
   - add focused regression coverage for approvals, timeline, diff, memory, and artifacts in PTY Claude mode

4. **Plan 04: PTY terminal coexistence regressions**
   - verify terminal stream/input/resize behavior still works
   - verify popup PTY mode still exposes terminal chat while keeping the other tabs reachable

## Risks And Pitfalls

- **Terminal-text temptation:** parsing PTY bytes to rebuild approvals or diffs is fragile and non-replayable
- **Live-only validation:** fixes that work with an already-open popup may still fail after reconnect
- **Workspace timing bugs:** memory endpoints can fail if session/workspace linkage is set too late
- **Artifact undercoverage:** `diff` and `artifacts` may diverge if only one gets tool-call fallback parity
- **PTY/Codex coupling:** avoid over-generalizing fixes if the real issue is Claude-specific

---

## Planner Guidance

- Keep plan count small and execution-oriented: 4 plans is enough here because terminal coexistence should not be mixed into the broader tab-parity plan
- Make the first plan TDD and contract-heavy
- Put daemon/runtime work before panel-level adjustments
- Require verification commands for both daemon and UI scopes
- Include explicit replay/reconnect acceptance criteria, not just “works while open”
