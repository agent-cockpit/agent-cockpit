# Phase 30.1: Claude PTY tab parity for approvals timeline diff memory and artifacts - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase fixes Claude-specific popup tabs after the move to the PTY runtime. The terminal itself now works through `node-pty`, but the non-terminal tabs still need to remain correct for PTY Claude sessions:

- `Approvals`
- `timeline`
- `diff`
- `memory`
- `artifacts`

The phase covers live updates, replay/catch-up behavior, daemon-to-UI contracts, and targeted UI/store adjustments required to keep those tabs accurate for PTY Claude sessions. It does not redesign the popup, replace the PTY transport, or broaden scope into Codex parity unless a shared abstraction is the cleanest way to fix Claude.

</domain>

<decisions>
## Implementation Decisions

### Source of Truth
- PTY terminal output is not the source of truth for the non-terminal tabs.
- `Approvals`, `timeline`, `diff`, `memory`, and `artifacts` must continue to be driven by normalized events and existing daemon endpoints, with replay-safe semantics.
- If PTY Claude currently bypasses structured events, fix that in the daemon/runtime contract first rather than adding UI logic that scrapes terminal text.

### Scope and Priority
- Prioritize Claude sessions launched through the new PTY path.
- Do not expand this phase into a generic popup redesign or broad provider refactor.
- Shared fixes are acceptable only when they directly reduce risk or duplication for the Claude PTY parity goal.

### Replay and Hydration
- Live behavior and replayed behavior must converge. Reopening a popup, reconnecting the WebSocket, or hydrating from REST endpoints must produce the same derived tab state as a continuously-open client.
- Session metadata needed by downstream tabs, especially `session_start` and workspace association, must be reliable for PTY Claude sessions.

### UI Strategy
- Reuse the existing panel components and store slices where possible.
- Prefer repairing event emission, session metadata, and hydration contracts over redesigning the tab UIs.

### Testing Strategy
- Planning should include both daemon-side and UI-side verification.
- The critical test shape is parity: PTY Claude session live updates plus replay/hydration after reconnect or reopen.

### the agent's Discretion
- Exact daemon module boundaries for PTY event derivation and session capability updates.
- Whether specific parity gaps are fixed by hook relay improvements, explicit runtime bookkeeping, DB/query changes, or minimal panel/store adjustments.
- Exact test file split and naming as long as parity is covered end-to-end.

</decisions>

<code_context>
## Existing Code Insights

### PTY Runtime Path
- `packages/daemon/src/adapters/claude/ptyLauncher.ts` launches Claude in a real PTY, configures hook relay settings, registers the Claude session id, and streams raw PTY output.
- `packages/daemon/src/ws/server.ts` exposes the PTY launch path, runtime registry integration, memory endpoints, session summary endpoints, and `pty_output` broadcasting.
- `packages/daemon/src/ws/handlers.ts` accepts `pty_input` and `pty_resize`, which are terminal-only side-channel messages and not normalized events.

### Structured Claude Events
- `packages/daemon/src/adapters/claude/hookServer.ts` and `packages/daemon/src/adapters/claude/hookParser.ts` remain the structured Claude event path for approvals, file changes, memory activity, and lifecycle synthesis.
- `packages/shared/src/events.ts` defines the normalized event contract that the non-terminal tabs depend on.

### UI Consumption Paths
- `packages/ui/src/hooks/useSessionEvents.ts` applies normalized events into Zustand and routes `pty_output` to the terminal bus separately.
- `packages/ui/src/components/office/InstancePopupHub.tsx` swaps the Chat surface to `TerminalPanel` when `liveSession.mode === 'pty'`, but keeps `ApprovalInbox`, `TimelinePanel`, `DiffPanel`, `MemoryPanel`, and `ArtifactsPanel` as-is.
- `packages/ui/src/components/panels/ApprovalInbox.tsx`, `TimelinePanel.tsx`, `DiffPanel.tsx`, `MemoryPanel.tsx`, and `ArtifactsPanel.tsx` all depend on normalized events and/or daemon REST hydration.

### Likely Fault Line
- The PTY work added a working raw terminal stream, but parity tabs still require reliable normalized event emission, session/workspace linkage, and replay-safe hydration.
- The most likely issue is not the terminal renderer itself; it is the structured event and metadata path that these tabs still require.

</code_context>
