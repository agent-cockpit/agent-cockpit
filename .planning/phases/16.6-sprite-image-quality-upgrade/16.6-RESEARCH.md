# Phase 16.6: Sprite Image Quality Upgrade — Research

**Researched:** 2026-04-12
**Domain:** Canvas 2D rendering quality, pixel art upscaling, PixelLab API resolution parameters
**Confidence:** HIGH

---

## Summary

Sprites look blurry at 2× zoom for two compounding reasons: (1) the browser's canvas 2D context applies bilinear smoothing by default when scaling, and (2) the source frames are stored at 64×64px even though the original PixelLab output was 92×92px (downscaled during sheet assembly). The canvas element in `OfficePage.tsx` has no `imageRendering: pixelated` CSS rule and no `ctx.imageSmoothingEnabled = false` call — either fix alone would eliminate the blur. Both fixes together are required because CSS affects the canvas element scaling while `imageSmoothingEnabled` controls `drawImage` scaling within the canvas.

There are two independent improvement axes for this phase:

**Axis A — Rendering fix (zero-cost, immediate):** Set `ctx.imageSmoothingEnabled = false` before all sprite `drawImage` calls, and add `image-rendering: pixelated` to the canvas element's inline style. This is the primary fix. It turns nearest-neighbor interpolation on for the 2× `ctx.scale()` zoom transform and for any sub-pixel scaling. This change alone will make the 64×64 sprites appear crisp.

**Axis B — Source resolution upgrade (PixelLab regeneration):** PixelLab supports up to 128×128px for `create_character`. All 10 characters were originally generated at 64px (PixelLab inflates to ~92px canvas output). Regenerating at 128px would yield ~128px actual character frames — double the pixel density. With nearest-neighbor rendering, 128px sprites would look identical or sharper at 2× zoom vs 64px sprites (both are displayed at 128 CSS px on screen at zoom=2). This is optional and high-cost (10 characters × all animation states × 8 directions = hundreds of API calls).

**Primary recommendation:** Implement Axis A first (rendering fix, 1 file change, instant result). Evaluate visually. If sprites remain unsatisfactorily small/low-detail even at nearest-neighbor, pursue Axis B for the astronaut only (player character, most visible). NPC characters are secondary — regenerate only if budget allows and visual inspection shows need.

## Root Cause Analysis

### Why sprites are blurry — confirmed (HIGH confidence)

**Cause 1: Missing `ctx.imageSmoothingEnabled = false`**

In `OfficePage.tsx` `render()`, the engine does:
```ts
ctx.scale(zoom, zoom)  // zoom = 2
ctx.drawImage(pImg, col * 64, row * 64, 64, 64, px, py, 64, 64)
```

The browser Canvas 2D API default is `ctx.imageSmoothingEnabled = true`, which applies bilinear filtering to all `drawImage` calls. When the canvas is scaled 2× via `ctx.scale()`, the 64×64 sprite source is effectively displayed at 128×128 logical pixels. With bilinear smoothing on, the browser interpolates between adjacent pixel values, producing blur. For pixel art, the correct interpolation is nearest-neighbor (`imageSmoothingEnabled = false`).

This affects: player sprite (OfficePage.tsx line 118) and NPC sprites (AgentSprite.tsx line 57).

**Cause 2: Missing `image-rendering: pixelated` on the canvas element**

The `<canvas>` in OfficePage has only `style={{ position: 'absolute', inset: 0, zIndex: 0 }}`. It does NOT have `image-rendering: pixelated`.

The CSS `image-rendering: pixelated` rule controls how the browser composites the canvas element itself into the page layout — relevant when the canvas's intrinsic size (set via `canvas.width`/`canvas.height`) differs from its CSS display size. If the canvas display size is scaled by CSS (e.g., DPR scaling on Retina displays), the browser would again apply bilinear filtering without this rule.

Note: `index.css` already has `image-rendering: pixelated` on `.agent-sprite` — but that class applies to CSS-rendered img elements from the old React sprite system, not to the game canvas.

**Cause 3 (contributing): Source resolution is 64×64px**

`build-spritesheet.ts` downsamples the 92×92px PixelLab output to 64×64px via `sharp.resize(fit: 'contain')`. So actual pixel data detail was 92×92 but is stored at 64×64. At 2× zoom (rendered at 128×128 CSS px), this means each original pixel becomes a ~2.78px square (128/46 — the character is ~46px tall in the 64×64 frame based on PixelLab's ~60% fill). This is acceptable for pixel art if nearest-neighbor is used; blurry only because of Cause 1+2.

### What does NOT cause the blur

- The `ctx.save()` / `ctx.restore()` wrapping the zoom transform — these are correct
- `drawImage` source coordinates — using 64×64 source slice is correct
- The TilemapRenderer `blit()` call — it correctly renders inside the zoom transform
- `WALK_FRAME_DURATION_MS` or frame selection logic — unrelated to sharpness

## Architecture Patterns

### Pattern 1: Set imageSmoothingEnabled once per render frame (RECOMMENDED)

Set `imageSmoothingEnabled = false` immediately after `ctx.save()`, before any draw calls. `ctx.save()` / `ctx.restore()` push/pop the full context state including `imageSmoothingEnabled`. Setting it once inside the save block covers all draws in that scope.

```typescript
// OfficePage.tsx — render() method, inside the engine class
render() {
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  const zoom = gameState.camera.zoom  // = 2

  ctx.save()
  ctx.imageSmoothingEnabled = false  // ADD THIS — nearest-neighbor for all draws below
  ctx.scale(zoom, zoom)

  // Layer 0: Tilemap
  tilemapRenderer.blit(ctx, gameState.camera.x, gameState.camera.y)

  // Layer 1: NPC sprites
  Object.values(liveSessions ?? {}).forEach((session) => {
    // ... drawAgentSprite calls
  })

  // Layer 2: Player sprite
  if (pImg?.complete && pImg.naturalWidth > 0) {
    ctx.drawImage(pImg, col * 64, row * 64, 64, 64, px, py, 64, 64)
  }

  ctx.restore()
}
```

### Pattern 4: Axis B — Regenerating at 128px via PixelLab REST API

If Axis A alone is visually insufficient, characters can be regenerated at 128px. The existing `character_id` for each character is reused — `animate_character` inherits the size from the stored character. However, the base character must be regenerated at 128px if the stored character was created at 64px.

**Important:** Per the PixelLab docs, `animate_character` inherits size from the character, not from parameters. To get 128px animations, the base character must have been created at `size=128`. The current 10 characters were created at `size=64` (evidence: 92×92px raw output, which matches PixelLab's ~1.44× canvas inflation of a 64px request). To get 128px animations, new base characters must be created with `size=128`, then all animations regenerated.

**Cost estimate for full 10-character Axis B:**
- 10 characters × 4 states (idle, blocked, completed, failed) × 1 call/state = 40 `animate_character` calls
- Plus walk state for astronaut = 1 more call
- Budget: 1438 images remaining (confirmed in PixelLab memory reference)
- Each `animate_character` call generates 8 directions of N frames each; total image count depends on frame counts (~4+8+9+7+8=36 frames × 8 dirs = 288 frames per character × 10 = 2880 frames)
- This would EXCEED the 1438 remaining budget for all 10 characters
- **Recommendation:** Axis B should be astronaut-only (player character) if pursued

### Pattern 5: Axis B — Build pipeline changes for 128px

```bash
# Change FRAME_SIZE in build-spritesheet.ts then rebuild:
# FRAME_SIZE = 128 (was 64)
npx tsx scripts/build-spritesheet.ts astronaut
# Produces: 1152×5120px astronaut-sheet.png (9 cols × 128 = 1152; 40 rows × 128 = 5120)
```

**drawImage call update if FRAME_SIZE changes to 128:**
```ts
// Was:   ctx.drawImage(img, col * 64, row * 64, 64, 64, px, py, 64, 64)
// After: ctx.drawImage(img, col * 128, row * 128, 128, 128, px, py, 64, 64)
// Note: dest size stays 64x64 — the sprite renders same screen size, just higher fidelity
// Alternatively: render at 128x128 dest to show double-size characters on screen
```

If Axis B is chosen, decide whether to ALSO double the on-screen render size (64→128 dest in `drawImage`). At 2× zoom, a 64px dest already maps to 128 CSS px. Doubling to 128px dest at 2× zoom = 256 CSS px per character — would be too large. Keep dest at 64×64 but use 128px source for higher fidelity (more detail visible per pixel).

### Anti-Patterns to Avoid

- **Setting `imageSmoothingEnabled` only in `AgentSprite.tsx` and not in `OfficePage.tsx`:** The player sprite is drawn directly in `OfficePage.tsx`, not through `AgentSprite`. Both call sites need the flag.
- **Confusing `imageSmoothingEnabled` scope with `ctx.save()`:** `ctx.save()` stacks the value; if you set it to `false` inside a `ctx.save()` block, it is restored to `true` on `ctx.restore()`. Always set it in the innermost scope where draws happen, or set it once outside the save block to persist globally.
- **Axis B without recreating base characters:** Existing `character_id` values generated at 64px cannot produce 128px animations via `animate_character` — the size is baked into the character creation. Regeneration requires new `create_character` calls.
- **Changing sprite sheet FRAME_SIZE without updating all drawImage call sites:** OfficePage.tsx and AgentSprite.tsx both hardcode `64` in their drawImage arguments. If FRAME_SIZE changes, both must be updated atomically.
- **Attempting Axis B for all 10 characters:** Would exceed the remaining image budget. Player character (astronaut) only if pursued.

## Common Pitfalls

### Pitfall 1: imageSmoothingEnabled resets on ctx.restore()
**What goes wrong:** Setting `ctx.imageSmoothingEnabled = false` inside a `ctx.save()` block, then drawing sprites outside the block — smoothing is back to `true`.
**Why it happens:** `ctx.save()` / `ctx.restore()` are a full state stack, not just for transforms.
**How to avoid:** Set `imageSmoothingEnabled = false` immediately after every `ctx.save()` call where pixel art draws occur, OR set it once before the `ctx.save()` block (it will be saved/restored but since you set it to `false` before the push, the post-restore state is also `false`).
**Recommended approach:** Set after `ctx.save()`, before `ctx.scale(zoom, zoom)` — this is the cleanest, most readable pattern.

### Pitfall 2: CSS image-rendering not supported in all browsers
**What goes wrong:** `image-rendering: pixelated` may render differently across browsers.
**Why it happens:** Browser compatibility: Chrome/Edge/Firefox all support `pixelated`. Safari uses `image-rendering: -webkit-optimize-contrast` or `crisp-edges`. Modern Safari (15.4+) supports `pixelated`.
**How to avoid:** Use `imageRendering: 'pixelated'` in React inline style. Add fallback `imageRendering: 'crisp-edges'` for older Safari if needed. For the target dev-tool audience (Chromium-based browsers are primary), `pixelated` alone is sufficient.

### Pitfall 3: Tilemap becomes blurry after imageSmoothingEnabled fix
**What goes wrong:** The map tileset was pre-rendered at 3232×3232 and contains fine details that look WORSE with nearest-neighbor at fractional scroll positions.
**Why it happens:** The tilemap is a photograph-quality pre-render; nearest-neighbor causes jagged edges at non-integer camera positions.
**How to avoid:** Set `ctx.imageSmoothingEnabled = false` for sprites only, and `ctx.imageSmoothingEnabled = true` for the tilemap blit. Structure the render to toggle the flag between layers.

**Correct render order:**
```ts
ctx.save()
ctx.scale(zoom, zoom)

// Tilemap — bilinear OK for pre-rendered composite
ctx.imageSmoothingEnabled = true
tilemapRenderer.blit(ctx, camX, camY)

// Sprites — nearest-neighbor required for pixel art
ctx.imageSmoothingEnabled = false
// ... drawAgentSprite calls ...
// ... player drawImage call ...

ctx.restore()
```

### Pitfall 4: Axis B — 128px sprites make characters appear physically larger
**What goes wrong:** If dest size in `drawImage` is changed from 64 to 128, sprites render at 128px destination, which at 2× zoom = 256 CSS px. Characters become huge on screen.
**Why it happens:** The destination rect in `drawImage(src, sx, sy, sw, sh, dx, dy, dw, dh)` determines on-screen size.
**How to avoid:** If regenerating at 128px source, keep `dw=64, dh=64` in `drawImage` — the 128px source frames are sampled into a 64px dest, yielding higher-density renders at the same physical size. This is the "retina-like" approach for pixel art.

### Pitfall 5: `imageSmoothingEnabled` not tested (no visual assertion in Vitest)
**What goes wrong:** The fix cannot be tested with Vitest canvas mocks — jsdom canvas stubs don't implement `imageSmoothingEnabled`.
**Why it happens:** jsdom does not implement canvas 2D API rendering faithfully.
**How to avoid:** The only valid test is manual visual QA. The Vitest test should verify the flag is SET (i.e., the property is assigned), not that the visual output is crisp. A simple spy on `ctx.imageSmoothingEnabled` setter in the test suite confirms the code path. The human-verify checkpoint is blocking for this phase.

## Code Examples

Verified patterns from OfficePage.tsx and Canvas 2D API:

### Fix: imageSmoothingEnabled + image-rendering (Axis A — complete)

```typescript
// OfficePage.tsx — render() method
render() {
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  const { sessions: liveSessions, events: liveEvents } = useStore.getState()
  const zoom = gameState.camera.zoom

  ctx.save()
  ctx.scale(zoom, zoom)

  // Tilemap first — bilinear smoothing OK for pre-rendered composite map
  ctx.imageSmoothingEnabled = true
  tilemapRenderer.blit(ctx, gameState.camera.x, gameState.camera.y)

  // Sprites — nearest-neighbor required for pixel art sharpness
  ctx.imageSmoothingEnabled = false

  // NPC sprites
  Object.values(liveSessions ?? {}).forEach((session) => {
    const pos = gameState.npcs[session.sessionId]
    if (!pos) return
    const sessionEvents = liveEvents[session.sessionId] ?? []
    const lastEvent = sessionEvents.at(-1)
    drawAgentSprite({
      ctx,
      session,
      lastEvent,
      position: { x: pos.x - gameState.camera.x, y: pos.y - gameState.camera.y },
      imageCache: imageCacheRef.current,
      tick: gameState.tick,
    })
  })

  // Player sprite
  const pImg = playerImgRef.current
  if (pImg?.complete && pImg.naturalWidth > 0) {
    const px = gameState.player.x - gameState.camera.x
    const py = gameState.player.y - gameState.camera.y
    const dirRow = DIRECTION_ROWS[gameState.player.direction as Direction] ?? 0
    const stateOffset = gameState.player.animTime > 0 ? STATE_ROW_OFFSET.walk : STATE_ROW_OFFSET.idle
    const row = dirRow + stateOffset
    const col = Math.floor(gameState.player.animTime / WALK_FRAME_DURATION_MS) % WALK_FRAME_COUNT
    ctx.drawImage(pImg, col * 64, row * 64, 64, 64, px, py, 64, 64)
  }

  ctx.restore()
}
```

## Open Questions

1. **Will Axis A alone be visually sufficient?**
   - What we know: `imageSmoothingEnabled = false` is the standard fix for this class of problem in all Canvas 2D pixel art games. The source frames at 64×64 are true pixel art (not photographs).
   - What's unclear: Whether the user's aesthetic expectation is "no blur" (Axis A suffices) or "more detail/larger apparent size" (Axis B needed).
   - Recommendation: Human-verify checkpoint after Axis A. Gate Axis B on the result.

2. **Does the tilemap also look blurry?**
   - What we know: TilemapRenderer blits a 3232×3232 pre-rendered composite. At 2× zoom, this is still bilinear-scaled.
   - What's unclear: Whether the map blur is acceptable (maps often look fine with bilinear) or also needs nearest-neighbor.
   - Recommendation: Keep tilemap on bilinear (`imageSmoothingEnabled = true`) and sprites on nearest-neighbor. Evaluate tilemap separately after sprites are fixed.

3. **Should NPC character sheets also be regenerated (Axis B)?**
   - What we know: All 10 NPC character sheets are 576×2048 at 64px frames. They were generated in Phase 13.
   - What's unclear: NPCs are smaller on screen (farther from camera); may be visually acceptable at 64px nearest-neighbor.
   - Recommendation: Out of scope for Phase 16.6. Address if NPC quality is raised as a separate concern.
